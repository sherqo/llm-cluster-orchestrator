The default strategy is least connections. This means the Master chooses the healthy worker with the lowest number of active requests. This is useful because LLM requests can take different amounts of time, so simply rotating requests evenly may overload one worker while another is idle.

The Master also tracks in-flight requests. Before sending a request to a worker, it adds the request to the in-flight store:

r.AddInFlight(requestID, worker.addr)

After the worker finishes, the request is removed:

r.RemoveInFlight(requestID)

This allows the dashboard to show which requests are currently being processed and how long they have been running.

The Master also runs a health check loop. Every five seconds, it pings workers using gRPC. If a worker responds successfully, it stays healthy. If it fails, the worker is marked as draining, which means it should no longer receive new requests.



6.3 Agent Implementation

The Agent is responsible for managing resources on a specific machine. It acts as the bridge between the Master and the local environment where workers run.

    Creates a Docker Manager.

    Registers HTTP endpoints.

    Registers itself with the Master.

    Registers itself with the Master.

6.4 Worker Implementation

The Worker is the component that processes user requests. It is implemented in Python and located in the /worker directory.

The main worker file is worker/main.py. Each worker runs a gRPC server and listens for requests from the Master. The worker implements the WorkerService defined in proto/worker.proto.

The worker provides two main gRPC methods:

Method


Purpose

Ping


Used by the Master to check if the worker is alive

Handle


Used to process chat requests

The Ping method simply returns:

worker_pb2.PingResponse(status="ok")

The Handle method performs the actual work. When a request arrives, the worker receives:

    request_id

    message

    priority

The worker stores requests in an in-memory priority queue implemented using Python’s heapq. This is intended to allow higher-priority requests, such as Pro users, to be handled before lower-priority Free users.

The general worker flow is:

    Receive gRPC request from Master.

    Determine request priority.

    Add request to priority queue.

    Pop the highest-priority request.

    Retrieve relevant context using RAG.

    Send the prompt and context to Ollama.

    Return the generated response to the Master.

The RAG retrieval is done using:

rag_context = retrieve(message)

The LLM inference is done using:

reply = run_model(prompt=message, context=rag_context, worker_port=WORKER_PORT)

Finally, the worker returns:

worker_pb2.Response(

    request_id=req_id,

    reply=reply,

    queue_length=depth,

)

This means each worker is not just a simple model server. It includes request prioritization, RAG retrieval, model inference, and queue-depth reporting.



6.5 gRPC Communication

gRPC is used for communication between the Master and Workers. The gRPC contract is defined in proto/worker.proto.

The service definition is:

service WorkerService {

  rpc Handle (Request) returns (Response);

  rpc Ping (PingRequest) returns (PingResponse);

}

The Handle RPC is used for normal chat processing. The Ping RPC is used for health checks.

The request message is:





message Request {

  string request_id = 1;

  string message = 2;

  int32 priority = 3;

}

The response message is:

message Response {

  string request_id = 1;

  string reply = 2;

  int32 queue_length = 3;

  int32 retries = 4;

}

On the Master side, Go gRPC code is generated inside master/generated/. The Master creates a gRPC connection to each worker in master/lib/worker.go using:

grpc.DialContext(...)

The connection uses keepalive settings so the Master can maintain a long-lived connection with workers.

When the Master sends a request to a worker, it calls:

w.client.Handle(ctx, &pb.Request{

    RequestId: requestID,

    Message:   req.Prompt,

    Priority:  tierToPriority(req.Tier),

})

When the Master checks worker health, it calls:

w.client.Ping(ctx, &pb.PingRequest{})

On the Worker side, Python generated files worker_pb2.py and worker_pb2_grpc.py are used. The worker registers itself as a gRPC server using:

worker_pb2_grpc.add_WorkerServiceServicer_to_server(worker, server)

This design separates the external HTTP API from the internal high-performance worker communication layer.



6.6 HTTP API Endpoints

HTTP is used for control-plane communication in the system. Control-plane communication means the requests that manage the cluster itself, such as registering agents, creating workers, notifying the Master that a worker is ready, and receiving user chat requests.

The system uses HTTP mainly in two places:

    Client to Master: the client sends chat requests to the Master.

    Master to Agent: the Master asks Agents to create or manage workers.

    Agent to Master: the Agent registers itself and notifies the Master when workers become ready.

The actual LLM task execution between the Master and Workers is not done through HTTP. That part uses gRPC. HTTP is used for coordination and management.

/chat Endpoint

The /chat endpoint is the main endpoint used by clients. It receives a user prompt, assigns it a request ID, forwards it to a suitable Worker, and returns the generated response.





Purpose

This endpoint acts as the external entry point of the LLM cluster. Instead of the client communicating directly with Workers, the client only communicates with the Master. The Master then decides which Worker should process the request.

Request Method

POST /chat

Request Body

The request body is decoded into the ChatRequest struct:

type ChatRequest struct {

    UserID string `json:"userId"`

    Prompt string `json:"prompt"`

    Tier   string `json:"tier"`

}

Example request:

{

  "userId": "user-1",

  "prompt": "What happened on D-Day?",

  "tier": "pro"

}

Field Explanation

Field


Description

userId


Identifies the user sending the request

prompt


The actual question or message sent by the user

tier


Represents the user type, such as pro or free

The tier field is important because it can be converted into a priority value before sending the request to a Worker. In the Master code, pro users are intended to receive higher priority than free users.

Internal Flow

When /chat receives a request, the Master performs these steps:

    Checks that the HTTP method is POST.

    Decodes the JSON request body.

    Records the request in the metrics system.

    Admission control is applied to avoid overload.

    Generates a unique request ID using UUID.

    Call the router to process the request.

    Returns the Worker response to the client.

In the code, the request ID is generated using:

requestID, err := uuid.NewV7()

Then the Master calls:

resp, err := router.HandleChat(r.Context(), requestIDStr, req)

This passes the request to the routing layer, which selects a Worker and sends the task using gRPC.

Response Body

The response is encoded using the ChatResponse struct:

type ChatResponse struct {

    RequestID string `json:"requestId"`

    Reply     string `json:"reply"`

}

Example response:

{

  "requestId": "generated-uuid",

  "reply": "Generated answer from the worker"

}
