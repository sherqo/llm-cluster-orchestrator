Events Tab

The Events tab shows important system notifications.

Events are created using:

monitoring.Event(place, msg)

Events are stored separately from normal logs. This makes the Events tab cleaner and focused on important cluster changes.

Examples of events include:

    Master server started.

    Agent registered.

    Worker registered through callback.

    Worker re-adopted after Agent restart.

    Autoscaler started.

    Scale-up or scale-down action occurred.

    Worker failed health check.

The Events tab is useful during demonstrations because it shows high-level system activity without all the lower-level debug details.





Metrics Layer

The metrics layer is implemented in:

master/lib/metrics.go

It collects and smooths cluster metrics. These metrics are used both by the dashboard and by the autoscaler.

The main metrics structure is:

type ClusterMetrics struct {

    QueueDepth        float64

    P95Latency        float64

    WorkerUtilization float64

    ErrorRate         float64

    RequestsPerSec    float64

    TotalWorkers      int

    HealthyWorkers    int

    InFlight          int

    QueueSize         int

    RPSTrend          float64

    CollectedAt       time.Time

}

The metrics include:

Metric


Description

Queue depth


Number of queued or pending requests

P95 latency


95th percentile request latency

Worker utilization


Average Worker load

Error rate


Percentage of failed requests

Requests per second


Current throughput

Total Workers


Number of Workers in the system

Healthy Workers


Number of Workers available for routing

In-flight


Number of active requests

Queue size


Current admission queue size

RPS trend


Whether traffic is increasing or decreasing

The metrics layer uses rolling windows and EWMA smoothing. EWMA stands for Exponentially Weighted Moving Average. It smooths noisy values so the dashboard and autoscaler do not overreact to short spikes.



Connection Between Monitoring and Autoscaling

The monitoring system is not only for display. It also supports autoscaling.

The autoscaler uses monitored metrics to decide whether to add or remove Workers. For example:

Condition


Possible Autoscaler Action

High worker utilization


Add more Workers

High P95 latency


Add more Workers

Queue depth increasing


Add more Workers

Low utilization for a sustained period


Remove Workers

High error rate


Avoid scale-down

Strong positive RPS trend


Prewarm Workers

This means the monitoring system is part of the control loop of the cluster. It observes the system, provides data to the autoscaler, and the autoscaler acts based on that data.

6.11 Code Structure

The project is organized into separate directories, where each directory represents a major component of the distributed LLM cluster. This structure makes the system easier to understand, maintain, and extend because each component has a clear responsibility.

The main project structure is:

Each part of the codebase contributes to a specific layer of the system.



master/ Directory

The master/ directory contains the Master node, which acts as the central controller of the cluster.

The Master is responsible for:

    Receiving user chat requests.

    Registering Agents.

    Registering Workers.

    Routing requests to Workers.

    Tracking in-flight requests.

    Monitoring Worker health.

    Running the autoscaler.

Displaying the TUI dashboard.
