The master selects the least-loaded worker and transitions it to DRAINING state. No new requests are routed to a draining worker. Once its in-flight requests complete (or the drain timeout elapses), the master sends a shutdown command and the worker exits cleanly.



7.5 Cooldowns and Soft Thresholds

Without cooldowns, the autoscaler can thrash — rapidly scaling up and down in response to transient spikes, which itself introduces latency and wastes resources. Two cooldown windows are enforced:



Cooldown Type


Purpose

Scale-up cooldown


Minimum time between consecutive scale-up events (e.g., 30 s). Prevents spawning many workers in response to a single burst before newly started workers have registered.

Scale-down cooldown


Minimum time between consecutive scale-down events (e.g., 60 s). Prevents removing workers too aggressively after a traffic drop.



Soft thresholds add hysteresis: scale-up fires at queue depth > 10 but scale-down only fires when queue depth = 0 (not just < 10). This prevents oscillation around the boundary.



7.6 Worker Lifecycle States

Each worker in the master's registry passes through a defined state machine:



State


Meaning


Receives Traffic?

STARTING


Worker process launched; awaiting registration + first health check


No

HEALTHY


Registered and passing health checks; normal operation


Yes

DRAINING


Marked for shutdown; completing in-flight requests only


No (new requests only)

DEAD


Health checks failed or drain timeout elapsed; removed from registry


No



The state transitions are: STARTING → HEALTHY (on successful registration + health check), HEALTHY → DRAINING (on scale-down selection), DRAINING → DEAD (on drain completion or timeout), HEALTHY → DEAD (on missed health checks).



7.7 Cost and Resource Efficiency

The autoscaler is designed to be cost-aware. Running LLMs on GPU hardware is expensive — every idle worker represents wasted spend. The scale-down logic is intentionally aggressive about removing idle capacity while the cooldowns prevent the cost of thrashing.



Efficiency Principle

The system targets a steady-state worker utilization of 60–80%. Below 40% utilization triggers scale-down consideration. Above 85% active concurrency triggers scale-up. This band prevents both waste and overload.





8. Fault Tolerance

A production LLM cluster must handle failures gracefully — workers crash, networks partition, and hardware fails. The LLM Cluster Orchestrator implements multiple layers of fault tolerance to ensure requests are not silently dropped and the system recovers automatically.

8.1 Worker Failure Detection

The master detects worker failures through a combination of missed heartbeats and active health checks. Every worker sends a periodic heartbeat to the master. If a worker misses a configurable number of consecutive heartbeats, the master marks it as DEAD and removes it from the routing pool.

Detection Mechanism


Detail

Heartbeat interval


Workers ping master every N seconds (configurable)

Miss threshold


After K consecutive missed heartbeats, worker transitions HEALTHY → DEAD

gRPC error detection


Immediate failure if the gRPC stream to worker errors (connection refused, EOF)

In-flight tracking


inflight.go tracks all outstanding requests per worker; on worker death, in-flight requests are surfaced for retry or error response

Screenshot of worker logs showing heartbeat messages being sent, or master logs showing health check results (PASS/FAIL) for each registered worker

8.2 Health Checks

Beyond heartbeats, the master performs active health checks by sending a lightweight ping RPC to each worker. This catches cases where the worker process is alive but the LLM inference engine has stalled

8.3 Request Timeout Handling

LLM inference can be slow. Without timeouts, a single slow model could hold a worker hostage indefinitely. The master enforces two timeout boundaries:

    Request timeout: Maximum time allowed for a complete request/response cycle. Exceeded requests return a 504 Gateway Timeout to the client.

    Drain timeout: Maximum time a DRAINING worker waits for in-flight requests to complete before being forcefully terminated.



8.4 Slow Worker Handling

A worker may be alive and healthy by heartbeat but processing requests abnormally slowly — for example, if the GPU is thermally throttling. The latency-based autoscaling trigger catches this: if p95 latency exceeds the SLO, the system scales out additional workers rather than waiting for the slow worker to finish.

The routing strategy (round-robin or least-connections) naturally routes new requests away from the overloaded worker while it finishes its current load.



8.5 What Happens if a Worker Dies Mid-Request

If a worker dies while processing a request, the in flight tracker in the master detects the orphaned request when the worker's gRPC stream closes. The master returns an error response to the client. The current implementation does not automatically retry mid-flight requests, as LLM inference is not idempotent retrying could produce duplicate completions.

Mid-request retries are not implemented. Clients are expected to implement retry logic with idempotency keys if needed.

8.6 What Happens if the Master Dies

The master is currently a single point of failure. If it crashes, workers continue running but cannot receive new routing assignments, and clients cannot submit new requests. The master is designed to restart quickly it persists state to SQLite so it can reconstruct the worker registry and in-flight state on restart.

The README notes that restart can be handled by systemd or the kernel watchdog. A highly-available master (with leader election) is a future enhancement.

Screenshot of master restart — show the server coming back up and re-registering workers from its persisted database state.

8.7 Updating Workers Without Downtime

Rolling updates are supported through the worker lifecycle. To deploy a new worker version:

    Start new workers with the updated binary — they register as STARTING then HEALTHY

    Drain old workers — transition them to DRAINING; they finish current requests

    Old workers exit after drain completes — zero dropped requests



This rolling approach means at least one old and one new worker overlap during the update window, ensuring continuous availability.

8.8 Current Fault Tolerance Limitations



Limitation


Impact / Mitigation

No mid-request retry


Client sees error if worker dies mid-inference; client must retry

Single master (no HA)


Master crash = full outage until restart; mitigated by fast restart + system

No cross-node agent coordination


Each agent manages one host; no cross-host failover of workers

No persistent request queue


Queued-but-not-started requests are lost on master crash

No circuit breaker


A misbehaving worker receives traffic until heartbeat timeout elapses
