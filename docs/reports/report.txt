Abstract 

Large Language Model services are expensive, slow compared with ordinary web requests, and sensitive to sudden bursts of traffic. A single inference server can become a bottleneck because each request may require token generation, retrieval of supporting context, and GPU or CPU computation. The project solves this problem by designing a distributed LLM cluster orchestrator that accepts many client requests, chooses suitable workers, executes inference in parallel, and monitors cluster health. 

 The implemented system is organized around a Go master/load balancer, Go agents that manage local worker containers, and Python worker nodes that expose a gRPC service. The worker integrates Retrieval-Augmented Generation through a Chroma-backed document store and uses Ollama as the local model runtime. The master provides HTTP endpoints for clients and agents, tracks in-flight work, applies admission control, records latency and error metrics, and runs an autoscaling loop. 

 The main distributed computing techniques used are request routing, least-connections scheduling, round-robin scheduling, bounded admission control, asynchronous metrics collection, worker lifecycle management, gRPC keepalive, retry-oriented routing, and autoscaling with cooldowns and hysteresis. Testing is supported by stress-test scripts that generate up to 1000 parallel requests and gradual bursts designed to observe autoscaling behavior. 

  

1. Introduction 

1.1 Project Background 

Modern LLM applications are often used by many users at the same time. Unlike a normal database lookup or static API call, an LLM request may take seconds because the system must prepare the prompt, optionally retrieve context, call a model runtime, and stream or return generated text. If all requests are sent to one server, the system becomes slow and fragile. Distributed computing addresses this by dividing the workload across multiple nodes and coordinating them through a controller. 

This project applies distributed-computing concepts to an LLM inference service. The system behaves like a small cluster: clients send chat requests to a master, the master routes each request to a worker, workers perform RAG and model inference, and agents help the master create or remove workers on available machines. 

1.2 Problem Statement 

The main problem is to serve 1000+ concurrent LLM requests with acceptable response time, controlled resource usage, and graceful behavior when workers are overloaded or unavailable. The system must distribute requests instead of letting one machine become the single computation point. It also needs a way to observe cluster load, add capacity when demand rises, and remove capacity when the system is idle. 

1.3 Project Objectives 

Design a distributed architecture for high-concurrency LLM inference. 

 • Implement a master/load balancer capable of routing requests to multiple workers.  

• Use gRPC for the internal hot path between the master and workers. 

 • Integrate RAG so answers can be grounded in an external knowledge base.  

• Track active requests, latency, throughput, errors, and worker state 

• Support autoscaling through agents that can spawn or terminate worker containers. 

• Evaluate the design using stress tests, failure simulation, and metrics. 

1.4 Scope of the Project 

The submitted implementation focuses on a working local/distributed prototype. It demonstrates request routing, worker registration, agent registration, worker creation through Docker, Chroma-based retrieval, Ollama-based generation, metrics collection, admission control, and autoscaling logic. It is not a fully production-hardened cloud platform, but it provides the core construction details needed to reason about such a platform. 

2. Background and Wider Reading 

2.1 Distributed Computing Concepts 

A distributed system is a collection of independent computers that cooperate over a network and appear to users as a single service. Instead of placing all work on one machine, computation is divided across nodes. Each node may run a process, store part of the data, execute tasks, or coordinate other nodes. This improves scalability and availability, but it also introduces communication delays, partial failures, and coordination problems.  

The most important idea in distributed computing is that failure is partial. In a single-process program, a crash is usually obvious. In a distributed system, one node can fail while others continue running, a network link can become slow, or a message can be delayed without being lost. Because of this, distributed systems usually need health checks, timeouts, retries, logging, and mechanisms for deciding which nodes are currently safe to use. 

Common distributed-computing terms include nodes, workers, coordinators, queues, replicas, messages, RPC calls, consistency, availability, and fault tolerance. A node is a machine or container that participates in the system. A worker performs actual computation. A coordinator assigns work or tracks state. Replication means keeping multiple copies of a service or data item to improve availability. A queue stores work temporarily when requests arrive faster than workers can process them. 

2.2 Load Balancing in Distributed Systems 

Load balancing is the process of distributing incoming requests across multiple backend nodes. The goal is to prevent one node from becoming overloaded while other nodes are idle. A good load balancer improves throughput, reduces average latency, and protects the service from sudden bursts of traffic.  

2.3 GPU Cluster Task Distribution 

GPU cluster task distribution refers to assigning compute-heavy jobs to a group of machines that contain GPUs or other accelerators. GPUs are effective for parallel numerical workloads because they can perform many matrix operations at the same time. Modern deep-learning inference and training workloads often depend on GPUs because neural networks require many tensor operations. 

Task distribution can be horizontal or parallel. Horizontal distribution sends different requests to different workers. Parallel distribution splits one large computation across multiple devices. Data parallelism processes multiple independent examples at the same time 

 

2.4 Large Language Model Inference Workloads 

LLM inference is the process of using a trained language model to generate text from an input prompt. Unlike a normal web request, an LLM request can be slow and  

LLM workloads are hard to balance because requests are not equal. A short question may finish quickly, while a long prompt with a long answer may use much more computation. The same service may also receive bursts of traffic, causing queues to form. This makes metrics such as active requests, queue depth, latency, throughput, and error rate important for controlling the system. 

Another important concept is the context window. The context window is the maximum amount of text that the model can consider at once. Longer context can improve answer quality when relevant information is included, but it also increases memory use and processing time. Systems that serve LLMs must therefore manage prompt size carefully. 

2.5 Retrieval-Augmented Generation 

Retrieval-Augmented Generation, called RAG, combines information retrieval with text generation. The main idea is to retrieve relevant external documents before asking the language model to answer. The retrieved text is placed into the prompt as context, so the model can generate an  answer using information that may not be stored in its parameters. The original RAG work describes this approach as a way to improve knowledge-intensive NLP tasks by combining parametric model knowledge with retrieved non-parametric knowledge 

2.6 Fault Tolerance and Autoscaling 

Fault tolerance means that a system continues to provide acceptable service even when some components fail. Distributed systems must expect failures because nodes can crash, containers can stop, networks can partition, disks can fill, and requests can time out. A fault-tolerant design assumes that failures will happen and limits their impact. 

Common fault-tolerance mechanisms include health checks, heartbeats, timeouts, retries, circuit breakers, replication, graceful shutdown, and request idempotency. A health check asks whether a component is alive and ready. A timeout prevents one slow operation from blocking forever. A retry attempts the operation again, usually on another node or after a delay. A circuit breaker stops sending requests to a component that is failing repeatedly. 

Autoscaling is the process of changing the amount of compute capacity based on demand. Horizontal autoscaling adds or removes replicas, while vertical scaling changes the resources assigned to existing replicas

3. Requirements Analysis 

3.1 Functional Requirements 

FR-1 - Client chat request handling. The system shall allow clients to submit LLM requests to the master through the HTTP POST /chat endpoint. A request contains userId, prompt, and tier fields. The master validates the JSON body, records the request in the metrics collector, generates a UUID-based request ID, forwards the request to an eligible worker, and returns a JSON response containing requestId and reply. 

FR-2 - Admission control and backpressure. The system shall protect the cluster from unlimited request bursts. The admission controller calculates the active concurrency limit from the number of healthy workers and a default maximum of 50 concurrent requests per worker. If the system is above the active limit, requests enter a bounded queue of up to 500 pending requests. If the queue is full, the master returns a service-unavailable response. If a request waits longer than 30 seconds, the master returns a queue-timeout response. 

FR-3 - Load-balanced worker selection. The master shall distribute accepted requests across registered workers. The default routing strategy is least-connections, where the router selects the healthy worker with the fewest active requests. The code also includes round-robin routing. Both strategies ignore non-routable workers and reject routing when the selected worker already has 50 or more active requests. 

FR-4 - Internal gRPC communication. The master shall communicate with workers using the WorkerService gRPC interface defined in worker.proto. The service exposes Handle for processing chat requests and Ping for health checks. A worker request contains request_id, message, and priority. A worker response contains request_id, reply, queue_length, and a retries field reserved in the protocol. 

FR-5 - Worker lifecycle management. The system shall maintain explicit worker lifecycle states. The code defines STARTING, WARMING, HEALTHY, DRAINING, STOPPING, and DEAD. Only HEALTHY workers are routable. STARTING workers are placeholders created after an agent accepts a spawn request, and they become usable only after the agent confirms readiness and the master opens a gRPC connection. 

FR-6 - Worker health checking. The master shall check worker availability. The health loop runs every 5 seconds and pings idle workers that are HEALTHY or WARMING. If Ping fails, the worker is marked DRAINING so that no new requests are routed to it. If Ping succeeds for a warming worker, the worker is marked HEALTHY. 

FR-7 - Agent registration. Agents shall register with the master through POST /agents/register. The registration includes agent_id, host, port, and optionally a list of running workers. This allows the master to know which agents are available and to re-adopt surviving workers after an agent or master restart. 

FR-8 - Docker-based worker creation. The master shall request new workers through an agent. The agent exposes POST /workers/create, creates a worker container from the configured image, assigns a free host port from the configured worker port range, labels the container with worker metadata, and runs it using host networking. The agent returns 202 Accepted and then waits for the worker gRPC port to become reachable before sending a /workers/ready callback to the master. 

FR-9 - Worker removal and cleanup. The system shall support worker destruction and cleanup through the agent. The agent exposes DELETE /workers/destroy and POST /workers/clean. The master can remove a worker from the router, close the gRPC connection, and request that the hosting agent remove the corresponding Docker container. 

FR-10 - Autoscaling. The system shall automatically add or remove workers based on observed load.  

FR-11 - Resource-aware agent selection. When multiple agents exist, the autoscaler shall select an agent for spawning using a least-workers-first policy. Failed spawn attempts are retried with exponential backoff, and agents with repeated failures are temporarily penalized so the autoscaler avoids immediately selecting them again. 

FR-12 - RAG context retrieval. Each worker shall attempt to retrieve supporting context from Chroma before calling the model. The worker loads documents from the configured Chroma collection, scores documents against prompt terms, selects the top documents, and adds them to the generated prompt. If Chroma is unavailable or no documents are loaded, the worker still proceeds without retrieved context. 

FR-13 - Ollama-based LLM inference. Each worker shall generate answers by sending a request to Ollama's /api/generate endpoint. The default model is smollm:135m. The worker builds either a context-augmented prompt or a direct prompt, disables streaming, limits generation length, applies a temperature value, and returns clear error strings for connection, timeout, HTTP, or unexpected failures. 

FR-14 - Priority-aware request handling. The system shall represent user tiers as priority values so that premium users can be served before free users. The worker contains a priority queue where lower numeric priority is served first. The intended behavior is pro-before-free processing, although the current master-to-worker numeric mapping needs correction as discussed in Section 3.5. 

FR-15 - Monitoring and dashboard support. The system shall expose enough runtime data for the terminal dashboard and debugging logs. The router and metrics collector track total workers, healthy workers, in-flight requests, queue size, request rate, error rate, P95 latency, smoothed utilization, autoscaler decisions, worker startup time, and recent in-flight/completed requests. 

FR-16 - Stress testing support. The project shall include scripts for evaluation. The stress test sends 1000 parallel requests to /chat with a mix of free and pro tiers. The gradual stress test sends phased bursts of 75, 150, 300, 50, and 500 requests to observe scaling behavior, queueing, cooldowns, and recovery. 

3.2 Non-Functional Requirements 

NFR-1 - Scalability. The system shall scale horizontally by creating additional worker containers instead of forcing all inference through one worker. Autoscaling is bounded by configurable minimum and maximum worker counts to prevent uncontrolled growth. 

NFR-2 - Responsiveness under load. The master shall use admission control, per-worker capacity limits, least-connections routing, and gRPC communication to reduce overload and keep requests responsive during bursts. 

NFR-3 - Fault tolerance at worker level. The system shall avoid routing new requests to workers that fail health checks or enter draining states. Failed or unreachable workers should be isolated from future traffic so healthy workers can continue serving requests. 

NFR-4 - Graceful scale-down. When removing workers, the autoscaler shall mark selected workers as DRAINING, wait for their active requests to complete for up to 120 seconds, then close the connection, mark the worker DEAD, remove it from the router, and ask the agent to destroy the container. 

3.3 User Stories 

    As an end user, I want to submit a prompt through the client or HTTP API so that I can receive an LLM-generated answer without knowing which worker handled the request. 

    As a free-tier user, I want my request to be accepted when the cluster has capacity, even if premium requests may receive higher priority during heavy load. 

    As a pro-tier user, I want the system to recognize my tier so that my requests can be prioritized when there is contention. 

    As a system administrator, I want to see worker count, agent count, active requests, queue depth, latency, error rate, and autoscaler activity so that I can understand the health of the cluster. 

    As the master, I want to admit requests only when capacity exists so that the cluster does not collapse under sudden bursts. 

    As the router, I want to select only HEALTHY workers so that requests are not sent to starting, draining, stopping, or dead workers. 

    As an agent, I want to register with the master and report surviving workers so that the master can rebuild its view of the cluster after restarts. 

    As an agent, I want to create worker containers and notify the master only after the worker port is reachable so that traffic is not routed to a worker before it is ready. 

    As a worker, I want to retrieve relevant Chroma documents before calling Ollama so that generated responses can use external context. 

    As the autoscaler, I want to add workers when utilization, latency, queue depth, or traffic trend shows sustained growth so that bursts can be handled. 

3.4 System Assumptions 

The master is the single entry point for client traffic and is reachable at the configured HTTP address, which defaults to localhost:8080 in the prototype. 

Clients communicate with the master, not directly with workers. This keeps worker addresses, lifecycle states, and routing strategy hidden from users. 

Agents are reachable from the master over HTTP and can advertise a host and port that the master can use for worker creation requests. 

Docker is installed and running on machines that host agents. The configured worker image is available before the agent attempts to create containers. 

Worker containers expose a gRPC service on the assigned port and use the protocol defined in worker.proto. 

Ollama is installed or reachable at the configured OLLAMA_URL. The default model smollm:135m is available or can be pulled before testing. 

Chroma is reachable at the configured CHROMA_URL when RAG is required. If Chroma is down, workers continue without RAG context. 

3.5 Constraints and Limitations from the Environment 

Local machine capacity.  limited by the CPU, memory, storage, Docker capacity, and Ollama performance of the local or LAN machines. The system demonstrates distributed orchestration, but it does not provide the same capacity as a real GPU cluster. 

Single-master limitation. The master is a single point of coordination for routing, metrics, admission control, worker registration, and autoscaling. If the master process fails, the current implementation depends on an external restart mechanism and does not automatically elect a new master. 

Host networking and port constraints. Worker containers use host networking and require available host ports. This is convenient for local development but can be less portable across operating systems and stricter container environments. 

Autoscaling delay. Autoscaling decisions are based on periodic metrics and worker-ready callbacks. Scale-up is not instant because the agent must create a Docker container, wait for the worker port, and notify the master. 

4. System Design and Architecture 

4.1 High-Level Architecture 

At a high level, the system consists of six main layers: the client layer, master/load balancer, agent, worker nodes, RAG/vector database, and Ollama LLM runtime. 

The client sends requests to the master using HTTP. The master acts as the system entry point and is responsible for selecting an appropriate worker. The worker receives the request using gRPC, optionally retrieves relevant context from the vector database, sends the final prompt to Ollama, and returns the response to the master. The master then returns the final response to the client. 

The agent operates beside the master as a resource manager. It is responsible for creating and destroying worker containers using Docker. This means worker lifecycle management is not hardcoded into the master. Instead, the master communicates with registered agents and asks them to create workers when more compute capacity is needed.

4.2 System Components 

    Client Layer 

The client layer represents the user-facing side of the system. In the implemented project, the client is a terminal-based chat interface written in Go. It allows a user to type prompts and receive responses from the distributed LLM system. 

The client communicates only with the master. It does not communicate with workers directly. This is important because the client should not need to know how many workers exist, where they are running, which workers are healthy, or which routing strategy is currently active. 

The client sends requests to the master using the /chat endpoint. A typical request has the following structure: 

{ 

  "userId": "somaya", 

  "prompt": "Who were the major leaders in World War II?", 

  "tier": "free" 

} 

The userId identifies the user, the prompt contains the question or task, and the tier field allows the system to apply priority rules. In the implementation, the master converts the tier into a priority value before forwarding the request to a worker. 

 

This design supports the coursework requirement of handling many concurrent user requests because the client layer is separated from the execution layer. Multiple clients can send requests to the same master endpoint, and the master is responsible for distributing those requests. 

 

    Master / Load Balancer 

The master is the central control component of the system. It is implemented in Go and acts as both a load balancer and cluster coordinator. 

The master exposes several HTTP endpoints: 

POST /chat 

POST /agents/register 

POST /workers/ready 

The /chat endpoint is used by clients to submit LLM requests. The /agents/register endpoint is used by agents when they start. The /workers/ready endpoint is used by agents to notify the master that a worker container has started successfully and is ready to receive gRPC requests. 

Inside the master, the Router object maintains the registered workers and agents. It is responsible for selecting a worker when a request arrives. The router also maintains an in-flight request tracker, which records which requests are currently being processed. 

The master performs the following functions: 

- Accepts incoming user requests. 

- Generates unique request IDs. 

- Applies admission control. 

- Selects workers using a routing strategy. 

- Sends requests to workers through gRPC. 

- Tracks in-flight requests. 

- Records latency and error metrics. 

- Registers agents and workers. 

- Performs health checks. 

- Runs the autoscaler. 

The master does not perform LLM inference itself. This is a deliberate design choice. If the master handled inference, it would become a computational bottleneck. Instead, it only coordinates the system and delegates computation to workers. 

The master also contains a terminal user interface that displays worker count, agent count, in-flight requests, logs, statistics, autoscaling state, and system events. This supports the administrator monitoring requirement in the coursework. 

    Agent 

The agent is a resource management service that runs on a compute node. It is responsible for managing workers’ containers on that machine. 

The agent is implemented in Go and exposes several HTTP endpoints: 

GET /system/info 

POST /workers/create 

DELETE /workers/destroy 

POST /workers/clean 

 

When the agent starts, it registers itself with the master by sending a request to: 

POST /agents/register 

The registration request includes the agent ID, host, port, and any already-running workers. This allows the master to know which agents are available for creating workers. 

When the master needs a new worker, it sends a request to the agent’s /workers/create endpoint. The agent then uses the Docker API to create a new worker container from the llm-worker:latest image. 

After creating the container, the agent does not immediately tell the master that the worker is ready. Instead, it waits until the worker’s gRPC port becomes reachable. Once the port is reachable, the agent sends a callback to: 

POST /workers/ready 

This callback-based design prevents the master from routing traffic to a worker before the worker is ready. 

The agent therefore acts as the bridge between the master’s scheduling logic and the machine-level Docker environment. 

 

    Worker Nodes 

Worker nodes are responsible for executing the actual LLM request. In this project, workers are implemented in Python and run inside Docker containers. 

Each worker exposes a gRPC service defined in the Protocol Buffers file worker.proto. The service includes two main RPC methods: 

rpc Handle(Request) returns (Response); 

rpc Ping(PingRequest) returns (PingResponse); 

 

The Handle method is used to process chat requests. The Ping method is used by the master health checker to verify that the worker is still alive. 

When a worker receives a request, it performs several steps: 

1. Receives the request from the master. 

2. Reads the request ID, message, and priority. 

3. Adds the request to its local priority queue. 

4. Retrieves relevant context using the RAG module. 

5. Builds a prompt using the user question and retrieved context. 

6. Sends the prompt to Ollama. 

7. Returns the generated response to the master. 

The worker uses a priority queue to support differentiated request handling. Higher-priority requests can be served before lower-priority requests. This supports the idea of user tiers such as free and pro users. 

Workers are designed to be disposable. If the system load increases, more workers can be created. If load decreases, extra workers can be drained and removed. This container-based design supports scalability. 

 

    RAG / Vector Database 

The RAG component improves the quality of responses by retrieving relevant information from a document collection before generating the answer. 

The project uses ChromaDB as the vector database. The vector database stores embedded documents and allows the worker to retrieve context related to the user’s query. 

 

The RAG logic is implemented in the worker through rag.py . When a prompt arrives, the worker attempts to load relevant documents from ChromaDB. The retrieved text is then inserted into the final prompt sent to Ollama. 

The RAG flow is: 

 

User question 

| 

Worker RAG module 

| 

Retrieve relevant context 

| 

Build augmented prompt 

| 

Send to Ollama 

 

This design is important for LLM systems because the model may not know all domain-specific information. By adding retrieved context, the system can provide more grounded and relevant answers. 

In the current implementation, RAG is optional. If ChromaDB is unavailable, the worker can continue and generate an answer using only LLM. This improves robustness during local testing. 

 

    Ollama / LLM Runtime 

Ollama is used as the local LLM inference runtime. It provides an HTTP API that workers can call to generate responses. 

The worker communicates with Ollama using the /api/generate endpoint. The model used is: 

 

The Ollama integration is implemented in worker/model.py. This module is responsible for: 

- selecting the model name, 

- reading the Ollama URL, 

- constructing the final prompt, 

 

- sending the HTTP request to Ollama, 

- handling timeouts and errors, 

- returning the generated text. 

The worker uses environment variables such as OLLAMA_URL to know where the Ollama service is running. This makes the deployment more flexible, because Ollama may run on the host machine, inside WSL, or on another server. 

Using Ollama allows the project to demonstrate real LLM inference without requiring a cloud API.

4.4 Request Lifecycle 

The request lifecycle describes what happens from the moment a user sends a prompt until a response is returned. 

First, the client sends an HTTP request to the master’s /chat endpoint. The master validates the JSON body and extracts the user ID, prompt, and tier. 

The master then records that a new request has arrived. This is used by the metrics collector to calculate request rate and error rate. 

Next, the master generates a unique request ID. This request ID allows the system to track the request across components. 

The master then calls the router to select a worker. The router only chooses workers that are currently healthy. Workers in STARTING, DRAINING, STOPPING, or DEAD states are not selected for new work. 

After selecting a worker, the master adds the request to the in-flight registry. This records that the request is currently assigned to a specific worker. 

The master then sends a gRPC request to the worker. The worker receives the request, processes it using RAG and Ollama, and returns a response. 

When the master receives the response, it removes the request from the in-flight registry and records the request latency. 

Finally, the master returns the response to the client.

4.5 Worker Registration Lifecycle 

Worker registration is a critical part of the system because the master must only route traffic to workers that are actually ready. 

The lifecycle begins when an agent starts. The agent sends a registration request to the master. This request includes the agent ID, host, port, and any running workers. 

The master stores this information in its agent registry. If the agent does not report any existing workers, the master requests that the agent creates one. 

The agent creates a worker container using Docker. At this stage, the worker is not immediately considered usable. The container may exist, but the Python gRPC server inside it may still be starting. 

To handle this, the master initially tracks the worker as STARTING. The agent then repeatedly checks the worker’s gRPC port. When the port becomes reachable, the agent sends a callback to the master’s /workers/ready endpoint. 

The master then tries to establish a gRPC connection to the worker. If the connection succeeds, the worker is registered as healthy and becomes available for request routing. 

The lifecycle is: 

Agent starts

4.6 Autoscaling Lifecycle 

The autoscaler is implemented inside the master. It periodically evaluates the cluster state and decides whether to add or remove workers. 

The autoscaler uses metrics collected from the router and metrics collector. These include: 

- total number of workers, 

- number of healthy workers, 

- number of in-flight requests, 

- queue depth, 

- worker utilization, 

- request latency, 

- error rate, 

- requests per second. 

The autoscaler supports scale-up and scale-down behavior. 

Scale-up happens when the system detects high load or when there are no workers available. The autoscaler checks whether the scale-up cooldown has passed, selects an available agent, and requests a new worker. 

Scale-down happens when the system detects sustained low load. The autoscaler does not immediately remove workers after a short quiet period. Instead, it waits for the low-load condition to remain true for a configured duration. This avoids unstable scaling behavior. 

During scale-down, workers are not killed immediately. A selected worker is first marked as DRAINING. This means it should not receive new requests, but it can finish any active requests. Once the worker has no active requests, it can be removed safely.

This lifecycle supports both performance and cost efficiency. The system can increase capacity when demand grows and reduce resource usage when demand is low. 

 

4.7 Failure Handling Design 

The system includes several mechanisms for handling failures. 

The first mechanism is worker health checking. The master periodically sends Ping requests to workers. If a worker fails to respond, it is marked as unhealthy or draining. This prevents the router from sending new requests to a failed worker. 

The second mechanism is request timeout handling. The master does not wait indefinitely for a worker response. If a worker becomes too slow or stops responding, the request can fail with a controlled error instead of blocking forever. 

The third mechanism is lifecycle-based routing. Workers are only considered routable when they are in the HEALTHY state. This prevents the system from sending requests to workers that are starting, draining, stopping, or dead. 

The fourth mechanism is autoscaler retry behavior. If worker creation fails, the autoscaler records the failure and uses cooldowns to avoid repeated rapid failures. 

The fifth mechanism is graceful scale-down. Workers are drained before removal, reducing the chance of interrupting active requests. 

Failure cases considered include: 

- worker container starts but gRPC server is not ready, 

- worker fails after registration, 

- worker becomes slow or overloaded, 

- agent fails to create a worker, 

- no workers are available, 

- request times out, 

- worker is removed during scale-down. 

There are still limitations. The master stores worker state and in-flight request state in memory. If the master crashes during a request, the current implementation may lose that request state. A production-grade version would require persistent queues, replicated masters, and durable request tracking.

5. Load Balancing and Scheduling Design 

5.1 Routing Problem Definition 

The routing problem in this system is to assign each incoming request to a worker that can process it quickly. This decision sits on the path of request handling, so the router must make a choice with low overhead while still considering worker health and current load. In practice, a worker is considered routable only if it is healthy and has not exceeded the configured concurrency threshold. This prevents the master from forwarding work to unavailable or saturated workers. 

Routing is more challenging for LLM workloads than for traditional web traffic because requests are not uniform. Some prompts are short and finish quickly, while others involve retrieval, long context windows, or extended token generation, which increases compute time significantly. If the router ignores this variability, one worker can become overloaded with expensive requests while others remain underused. Therefore, the design aims to balance three goals at once: fast decision making, fair distribution of work, and protection against overload. 

Routing Requirement 
	

Why It Matters 

Low-latency decision 
	

The master must choose a worker immediately so routing does not become a bottleneck itself. 

Health awareness 
	

Unhealthy or draining workers should not receive new traffic, which improves reliability. 

Load awareness 
	

Workers with too many active requests should be avoided to reduce queueing delay and timeouts. 

Fairness 
	

Requests should be spread across available workers so cluster resources are used efficiently. 

5.2 Implemented Strategies 

The router currently supports effective strategies that are suitable for a distributed inference system. Each strategy offers a different trade off between implementation simplicity, fairness, and sensitivity to workload imbalance. The two most relevant choices for this project are least connections and round robin.

The default router behavior uses least connections because it better reflects actual pressure on the cluster. Inference requests can vary widely in prompt length, retrieval cost, and generation length, so counting active work is a more useful signal than simply rotating evenly. The implementation scans all workers, skips any worker that is not routable, and chooses the one with the minimum active request count. It also applies a hard guardrail by rejecting routing if the chosen worker has already reached the configured concurrency ceiling, which helps avoid runaway overload.

5.3 Strategy Comparison 

Admission control complements routing by deciding whether the system should accept more work at all. Even the best routing algorithm cannot help if every healthy worker is already busy. For that reason, the master estimates total safe capacity based on the number of healthy workers multiplied by the per-worker concurrency limit. Requests beyond that capacity are placed in a bounded queue, giving the system a short buffer during traffic bursts without allowing memory usage or waiting time to grow without limit.

This design improves resilience under heavy load. Instead of silently degrading until response times become unacceptable, the system sheds excess load in a controlled and transparent way. That behavior is especially important for interactive AI systems, because users benefit more from a clear overload message than from an extremely delayed or failed response after a long wait. 

5.4 Why Each Strategy Matters 

Priority handling introduces a lightweight quality-of-service mechanism into the worker. Requests are placed into an in-memory priority queue where premium users receive priority 0 and free-tier users receive priority 1. Because lower numeric values are processed first, premium requests are served ahead of free requests whenever both classes are waiting at the same time. This approach does not change the routing destination directly, but it improves perceived responsiveness for high-value traffic once requests arrive at a worker.

5.5 Dynamic Strategy Selection Under Load 

The coursework also considered several strategy extensions that are not yet enabled in the main router but are valuable for future improvement. These strategies would make the load balancer more adaptive to real-time system behavior, user identity, or session locality. They are especially useful if the cluster grows larger or if the workload becomes more heterogeneous over time. 

Future Strategy 
	

Core Idea 
	

Potential Benefit 

Random Routing 
	

Selects a healthy worker at random. 
	

Very low overhead and useful as a simple baseline for testing. 

Least Response Time 
	

Routes to the worker with the best recent latency measurements. 
	

Avoids persistently slow workers and adapts to performance degradation.

Among these options, least response time and priority-aware routing are the most practical next steps for this project. The first would let the system react to observed performance rather than only active request counts, while the second would move service differentiation earlier in the processing pipeline. Together, they would make the scheduling layer smarter, more responsive, and better aligned with real production-style requirements.

6.2 Master Node Implementation 

The Master node is the central coordinator of the entire system. It is implemented in Go and is responsible for receiving client requests, selecting workers, forwarding requests to workers, tracking request state, monitoring worker health, and managing autoscaling. 

The Master starts from master/main.go. In this file, the system first creates a router using: 

router := lib.NewRouter() 

The router is the main in memory structure that stores the current cluster state. After creating the router, the Master starts the HTTP server using: 

go lib.Serve(router) 

Then it starts the terminal dashboard using: 

tui.Run(router) 

This means the Master runs two major parts at the same time: 

    The HTTP server that receives API requests. 

    The TUI dashboard that shows live cluster status. 

The main HTTP server implementation is in master/lib/server.go. The Master exposes endpoints such as /chat, /agents/register, and /workers/ready. 

The most important endpoint is /chat. When a client sends a chat request, the Master performs the following steps: 

    Validates that the request method is POST. 

    Parses the JSON request body into a ChatRequest. 

    Records the request in the metrics collector. 

    Applies admission control to avoid accepting too many requests during overload. 

    Generates a unique request ID using UUID. 

    Send the request to the router. 

    Returns the worker response to the client. 

The request format is represented by this Go struct: 

type ChatRequest struct { 

    UserID string `json:"userId"` 

    Prompt string `json:"prompt"` 

    Tier   string `json:"tier"` 

} 

The response is represented as: 

type ChatResponse struct { 

    RequestID string `json:"requestId"` 

    Reply     string `json:"reply"` 

} 

The actual routing happens in master/lib/router.go. The router stores workers in a map: 

workers map[string]*Worker 

It also stores registered agents: 

agents map[string]*AgentInfo 

This allows the Master to know which workers are available and which agent owns each worker. 

The Master supports multiple worker selection strategies. In master/lib/strategy.go, two strategies are defined: 

StrategyRoundRobin 

StrategyLeastConnections 

The default strategy is least connections. This means the Master chooses the healthy worker with the lowest number of active requests. This is useful because LLM requests can take different amounts of time, so simply rotating requests evenly may overload one worker while another is idle. 

The Master also tracks in-flight requests. Before sending a request to a worker, it adds the request to the in-flight store: 

r.AddInFlight(requestID, worker.addr) 

After the worker finishes, the request is removed: 

r.RemoveInFlight(requestID) 

This allows the dashboard to show which requests are currently being processed and how long they have been running. 

The Master also runs a health check loop. Every five seconds, it pings workers using gRPC. If a worker responds successfully, it stays healthy. If it fails, the worker is marked as draining, which means it should no longer receive new requests.

6.4 Worker Implementation 

The Worker is the component that processes user requests. It is implemented in Python and located in the /worker directory. 

The main worker file is worker/main.py. Each worker runs a gRPC server and listens for requests from the Master. The worker implements the WorkerService defined in proto/worker.proto. 

The worker provides two main gRPC methods: 

Method 
	

Purpose 

Ping 
	

Used by the Master to check if the worker is alive 

Handle 
	

Used to process chat requests 

The Ping method simply returns: 

worker_pb2.PingResponse(status="ok") 

The Handle method performs the actual work. When a request arrives, the worker receives: 

    request_id 

    message 

    priority 

The worker stores requests in an in-memory priority queue implemented using Python’s heapq. This is intended to allow higher-priority requests, such as Pro users, to be handled before lower-priority Free users. 

The general worker flow is: 

    Receive gRPC request from Master. 

    Determine request priority. 

    Add request to priority queue. 

    Pop the highest-priority request. 

    Retrieve relevant context using RAG. 

    Send the prompt and context to Ollama. 

    Return the generated response to the Master. 

The RAG retrieval is done using: 

rag_context = retrieve(message) 

The LLM inference is done using: 

reply = run_model(prompt=message, context=rag_context, worker_port=WORKER_PORT) 

Finally, the worker returns: 

worker_pb2.Response( 

    request_id=req_id, 

    reply=reply, 

    queue_length=depth, 

) 

This means each worker is not just a simple model server. It includes request prioritization, RAG retrieval, model inference, and queue-depth reporting. 

 

6.5 gRPC Communication 

gRPC is used for communication between the Master and Workers. The gRPC contract is defined in proto/worker.proto. 

The service definition is: 

service WorkerService { 

  rpc Handle (Request) returns (Response); 

  rpc Ping (PingRequest) returns (PingResponse); 

} 

The Handle RPC is used for normal chat processing. The Ping RPC is used for health checks. 

The request message is: 

 

 

message Request { 

  string request_id = 1; 

  string message = 2; 

  int32 priority = 3; 

} 

The response message is: 

message Response { 

  string request_id = 1; 

  string reply = 2; 

  int32 queue_length = 3; 

  int32 retries = 4; 

} 

On the Master side, Go gRPC code is generated inside master/generated/. The Master creates a gRPC connection to each worker in master/lib/worker.go using: 

grpc.DialContext(...) 

The connection uses keepalive settings so the Master can maintain a long-lived connection with workers. 

When the Master sends a request to a worker, it calls: 

w.client.Handle(ctx, &pb.Request{ 

    RequestId: requestID, 

    Message:   req.Prompt, 

    Priority:  tierToPriority(req.Tier), 

}) 

When the Master checks worker health, it calls: 

w.client.Ping(ctx, &pb.PingRequest{}) 

On the Worker side, Python generated files worker_pb2.py and worker_pb2_grpc.py are used. The worker registers itself as a gRPC server using: 

worker_pb2_grpc.add_WorkerServiceServicer_to_server(worker, server) 

This design separates the external HTTP API from the internal high-performance worker communication layer.

6.7. RAG Pipeline Implementation 

The system includes a Retrieval-Augmented Generation, or RAG, pipeline. The purpose of this pipeline is to improve the quality of LLM answers by giving the model extra information from an external document collection. 

Without RAG, the LLM would answer only using its pre-trained knowledge. With RAG, the Worker first searches for a local knowledge base, retrieves relevant text, adds that text to the prompt, and then sends the improved prompt to the LLM. 

In this project, the RAG pipeline is built using:

Purpose of the RAG Pipeline 

The RAG pipeline solves one important problem: the model may not always have enough accurate or project-specific knowledge to answer a question. Instead of depending only on the model’s internal knowledge, the system provides the model with relevant documents before generating the answer. 

For example, if the user asks: 

What happened on D-Day? 

The Worker can retrieve text from the embedded document about D-Day, add it to the prompt, and then ask the LLM to answer using that context. 

This makes the answer more grounded in the project’s own data.

Loading Documents in the Worker 

When the Worker starts, worker/rag.py immediately calls: 

DOCUMENTS = _load_documents() 

This means the Worker loads documents once at startup and stores them in memory. 

The _load_documents() function first checks whether the Chroma collection exists: 

col_resp = requests.get( 

    f"{CHROMA_URL}/api/v1/collections/{COLLECTION_NAME}", 

    timeout=5, 

) 

If the collection is not found, RAG is disabled: 

print("[rag] WARNING: collection not found - RAG disabled") 

return [] 

If the collection exists, the Worker gets the collection ID: 

col_id = col_resp.json()["id"] 

Then it requests the documents from ChromaDB: 

docs_resp = requests.post( 

    f"{CHROMA_URL}/api/v1/collections/{col_id}/get", 

    json={"include": ["documents"]}, 

    timeout=10, 

) 

The returned documents are stored in the global DOCUMENTS variable. 

This means the Worker does not query ChromaDB for every request in the current implementation. Instead, it loads the documents from ChromaDB once and performs retrieval locally in memory. 

 

Retrieval Function 

The main retrieval function is: 

retrieve(prompt) 

This function receives the user prompt and returns a context string that will later be added to the LLM prompt. 

The function first checks if any documents were loaded: 

if not DOCUMENTS: 

    return "" 

If no documents are available, it returns an empty string. In that case, the LLM still answers, but without RAG context. 

Next, the prompt is converted into search terms: 

terms = [t for t in prompt.lower().split() if len(t) > 2] 

This removes very short words and keeps only meaningful terms. 

For example, if the prompt is: 

What happened on D-Day? 

The terms may include: 

what 

happened 

d-day 

Each document is then scored using the _score() function: 

def _score(doc: str, terms: list[str]) -> int: 

    text = doc.lower() 

    return sum(text.count(t) for t in terms if t) 

This function counts how many times the prompt terms appear in each document. 

Documents are sorted by score: 

scored = sorted( 

    ((doc, _score(doc, terms)) for doc in DOCUMENTS), 

    key=lambda x: x[1], 

    reverse=True, 

) 

The highest-scoring documents are selected first. 

 

Selecting Top Context 

After scoring the documents, the retrieval function selects the top results: 

top_docs = [doc[:MAX_CHUNK_CHARS] for doc, s in scored[:top_k] if s > 0] 

This means: 

    Only the top k documents are selected. 

    Documents must have a score greater than zero. 

    Each selected document is truncated to MAX_CHUNK_CHARS. 

By default: 

RAG_TOP_K = 2 

RAG_MAX_CHARS = 150 

So, the Worker retrieves two short pieces of context, each up to 150 characters. 

If no document matches the prompt terms, the system falls back to the first documents: 

if not top_docs: 

    top_docs = [doc[:MAX_CHUNK_CHARS] for doc in DOCUMENTS[:top_k]] 

Finally, the selected context is formatted: 

context = "\n\n".join(f"[{i+1}] {doc}" for i, doc in enumerate(top_docs)) 

The result may look like: 

[1] D-Day was the Allied invasion of Normandy on June 6, 1944... 

 

[2] The Allied Powers coordinated military operations across Europe... 

This context is returned to the Worker. 

 

Adding RAG Context to the LLM Prompt 

After the Worker retrieves context, worker/main.py sends it to the model layer: 

rag_context = retrieve(message) 

reply = run_model(prompt=message, context=rag_context, worker_port=WORKER_PORT) 

The prompt construction happens in: 

worker/model.py 

The function responsible is: 

build_prompt(prompt, context) 

If context exists, the final prompt becomes: 

Use this context to answer the question. Give a direct answer only. 

Context: [retrieved context] 

Question: [user prompt] 

Answer: 

If no context is available, the prompt becomes: 

Answer this question directly: [user prompt] 

Answer: 

This design allows the system to continue working even if the vector database is unavailable. In that case, it simply falls back to normal LLM inference.

Fault Tolerance in the RAG Pipeline 

The RAG implementation includes basic fault tolerance. 

If the Chroma collection cannot be found, the Worker prints: 

[rag] WARNING: collection not found - RAG disabled 

If there is an error connecting to ChromaDB, the Worker catches the exception and returns an empty document list: 

except Exception as e: 

    print(f"[rag] WARNING: error loading documents: {e}") 

    return [] 

If DOCUMENTS is empty, the retrieval function returns: 

return "" 

This means the Worker does not crash if the vector database is unavailable. Instead, it continues to answer using only the LLM. 

 

Important Implementation Note 

Although the project uses ChromaDB as the vector database, the current Worker retrieval logic does not perform a semantic vector search for every prompt. Instead, it loads documents from ChromaDB once and then scores them locally using keyword frequency. 

So the current implementation is a hybrid/simple RAG approach: 

    ChromaDB stores and serves the document collection. 

    The Worker loads the documents from ChromaDB. 

    The Worker performs local keyword-based ranking. 

    The selected text is added to the LLM prompt. 

This is simpler than full semantic retrieval, but it still demonstrates the RAG concept clearly: retrieve external context first, then generate an answer using that context. 

 

6.8 LLM Inference Integration 

The system uses Ollama as the local LLM runtime. Ollama is responsible for running the language model and generating the final answer after the Worker prepares the prompt. 

The LLM integration is implemented mainly in: 

worker/model.py 

The Worker does not generate text by itself. Instead, it prepares the input prompt, sends it to Ollama through an HTTP API call, receives the generated response, and returns that response back to the Master through gRPC. 

 

Purpose of LLM Inference Integration 

The purpose of this part is to connect the Worker with a real language model. The Worker handles request management, RAG retrieval, and prompt formatting, while Ollama handles the actual text generation. 

This separation makes the system cleaner because: 

Component 
	

Responsibility 

Worker 
	

Receives request, retrieves context, builds prompt 

Ollama 
	

Runs the LLM and generates the answer 

Master 
	

Routes requests and returns responses to the client 

 

So, the Worker acts as a bridge between the distributed cluster system and the local LLM runtime. 

 

Default Model Configuration 

In worker/model.py, the default model is defined as: 

OLLAMA_MODEL = os.getenv("OLLAMA_MODEL", "smollm:135m") 

This means the system uses: 

smollm:135m 

as the default model if no other model is provided. 

However, the model can be changed without editing the code by setting the environment variable: 

OLLAMA_MODEL 

For example: 

OLLAMA_MODEL=llama3.2 

This makes the Worker flexible because different models can be tested or deployed depending on available hardware and performance requirements. 

The file also defines a timeout value: 

OLLAMA_TIMEOUT_SECONDS = int(os.getenv("OLLAMA_TIMEOUT_SECONDS", "120")) 

By default, the Worker waits up to 120 seconds for Ollama to respond. This is important because LLM inference can take time, especially on slower machines or with larger models.

The actual model inference happens in: 

run_model(prompt, context, worker_port) 

This function performs three main tasks: 

    Find the correct Ollama URL. 

    Builds the final prompt. 

    Send the prompt to Ollama. 

The Ollama URL is selected using: 

ollama_url = get_ollama_url(worker_port)

full_prompt = build_prompt(prompt, context)

        "model": OLLAMA_MODEL, 

        "prompt": full_prompt, 

        "stream": False, 

        "options": { 

            "temperature": 0.2,

Ollama Request Parameters 

The request body sent to Ollama contains several important fields. 

{ 

  "model": "smollm:135m", 

  "prompt": "final prompt", 

  "stream": false, 

  "options": {

    "num_predict": 200

    model 

This specifies which Ollama model should generate the answer. 

"model": "smollm:135m" 

In this project, the default model is lightweight, which helps make local inference faster. 

    prompt 

This contains the final prompt created by the Worker. It may include RAG context or may simply contain the user question. 

"prompt": "final prompt" 

    stream 

This is set to: 

"stream": false 

This means Ollama returns the complete answer in one response instead of streaming partial tokens. This makes the Worker implementation simpler because it waits for one complete JSON response. 

    temperature 

The temperature is set to: 

"temperature": 0.2 

A low temperature makes the output more stable and deterministic. This is useful for factual question answering because the system should avoid highly creative or random responses. 

    num_predict 

The response length is limited using: 

"num_predict": 200 

This limits the number of generated tokens. It helps reduce latency and prevents the model from producing unnecessarily long answers. 

 

Processing the Ollama Response 

After Ollama responds, the Worker checks for HTTP errors: 

response.raise_for_status() 

If the response is successful, the Worker extracts the generated text: 

return response.json().get("response", "").strip() 

Ollama returns the model output inside the response field. The Worker strips extra whitespace and returns the final answer to worker/main.py. 

Then worker/main.py wraps the result in a gRPC response: 

return worker_pb2.Response( 

    request_id=req_id, 

    reply=reply, 

    queue_length=depth, 

) 

This response is sent back to the Master. 

 

Error Handling 

The Worker includes error handling so that failures in Ollama do not crash the Worker process. 

    Connection Error 

If Ollama is not running or cannot be reached: 

except requests.exceptions.ConnectionError:

This returns a clear message instead of throwing an exception. 

    Timeout Error 

If Ollama takes too long to respond: 

except requests.exceptions.Timeout: 

    return f"Ollama timed out — try a smaller model" 

This is useful because larger models may be too slow for the available machine. 

    HTTP Error 

If Ollama responds with an error status: 

except requests.exceptions.HTTPError as exc: 

    return f"Ollama error: {exc} — {response.text}" 

This helps expose model-server errors during debugging. 

    General Error 

Any other unexpected error is caught using: 

except Exception as exc: 

    return f"Error: {exc}" 

This protects the Worker from crashing due to unexpected runtime issues.

6.9 Docker-Based Worker Deployment 

The system deploys each Worker as a separate Docker container. This is an important design choice because Workers are the execution units that handle LLM requests, RAG retrieval, and gRPC communication with the Master. By running Workers inside containers, the system can create, stop, and remove them dynamically without manually starting Python processes. 

This makes the cluster more flexible and scalable. When the system has more traffic, the Master can ask an Agent to create additional Worker containers. When the system has less traffic, unnecessary Workers can be removed. 

 

Purpose of Docker-Based Deployment 

Docker is used to make Worker deployment more reliable and repeatable. 

Instead of depending on the user to manually install Python packages and start each Worker, the Worker environment is packaged into a Docker image. Every Worker container starts from the same image and runs the same Python application.

Running containers can be rediscovered after restart 

 

Worker Docker Image 

The Worker Docker image is defined in: 

worker/Dockerfile 

The Dockerfile uses a lightweight Python image: 

FROM python:3.11-slim 

This means each Worker container runs on Python 3.11 but avoids unnecessary operating system packages, keeping the image smaller. 

The working directory inside the container is set to: 

WORKDIR /app 

Then the dependency file is copied: 

COPY requirements.txt . 

The dependencies are installed using: 

RUN pip install --no-cache-dir --retries 5 --timeout 120 -r requirements.txt 

The --no-cache-dir option reduces image size by avoiding pip cache files. The retry and timeout options make dependency installation more tolerant of temporary network delays during image build. 

The Dockerfile also sets: 

ENV PYTHONUNBUFFERED=1 

This ensures Python logs are printed immediately instead of being buffered. This is useful for debugging Worker containers because logs appear in real time. 

Finally, the Worker source code is copied: 

COPY . . 

The container starts the Worker using: 

CMD ["python", "main.py"] 

This means when the container starts, it runs worker/main.py, which starts the gRPC Worker server. 

 

Agent Responsibility in Worker Deployment 

The Agent is responsible for creating and managing Worker containers. This logic is implemented in: 

agent/docker.go 

The Agent uses the Docker Go SDK to communicate with the local Docker daemon. This is done in NewDockerManager(...): 

client.NewClientWithOpts( 

    client.FromEnv, 

    client.WithAPIVersionNegotiation(), 

) 

This allows the Agent to create, start, list, and remove Docker containers programmatically. 

The Agent creates Workers through: 

CreateWorker(req) 

This function is called when the Master sends a request to the Agent’s /workers/create endpoint.

    Prepares environment variables.

Port Allocation 

Each Worker runs a gRPC server and needs a port. The Agent manages a range of possible Worker ports. 

The port range is configured in AgentConfig: 

WorkerPortStart int 

WorkerPortEnd   int 

By default, these are loaded from command-line flags in agent/utils.go: 

flag.IntVar(&cfg.WorkerPortStart, "worker-port-start", 50051, "first host port to publish worker gRPC") 

flag.IntVar(&cfg.WorkerPortEnd, "worker-port-end", 50150, "last host port to publish worker gRPC") 

So, by default, the Agent can create Workers using ports from: 

50051 to 50150 

The Agent chooses a free port using: 

allocateHostPort() 

This function loops through the configured range and checks whether each port is available. 

The check is done using: 

net.Listen("tcp", fmt.Sprintf(":%d", port)) 

If the port can be opened, it is considered available. The Agent then closes it and assigns it to the new Worker. 

This prevents two Workers from trying to use the same gRPC port. 

 

Worker ID and Container Name 

After selecting a port, the Agent generates a unique Worker ID: 

workerID := fmt.Sprintf( 

    "worker-%s-%d-%d", 

    d.cfg.AgentID, 

    hostPort, 

    time.Now().UnixMilli(), 

) 

This ID contains: 

    The Agent ID. 

    The assigned Worker port. 

    A timestamp in milliseconds. 

This makes Worker IDs unique even when multiple Workers are created by the same Agent. 

The Docker container name is then created as: 

name := "llm-" + workerID 

For example, a Worker container name may look like: 

llm-worker-agent-192.168.1.10-9000-50051-1710000000000 

This naming convention helps identify which containers belong to the LLM cluster. 

 

Environment Variables Passed to Workers 

The Agent passes configuration to each Worker through environment variables. 

In agent/docker.go, the environment list is built like this: 

env := append([]string{}, req.Env...) 

env = append(

    fmt.Sprintf("WORKER_PORT=%d", hostPort), 

    "MASTER_URL="+d.cfg.MasterURL, 

) 

If Ollama and Chroma URLs are configured, they are also added: 

if d.cfg.OllamaURL != "" { 

    env = append(env, "OLLAMA_URL="+d.cfg.OllamaURL) 

} 

 

if d.cfg.ChromaURL != "" { 

    env = append(env, "CHROMA_URL="+d.cfg.ChromaURL) 

}

worker_port_env = os.getenv("WORKER_PORT") 

If WORKER_PORT exists, it uses that value. Otherwise, it falls back to a command-line argument or default port 50051. 

This allows the same Docker image to run many Worker containers on different ports. 

 

Docker Container Creation 

The Agent creates the Docker container using: 

d.client.ContainerCreate(...) 

The container config includes: 

&container.Config{ 

    Image: image, 

    Env:   env, 

    Labels: map[string]string{ 

        "llm.cluster.role":      "worker",

This tells Docker: 

    Which image to use. 

    Which environment variables to pass. 

    Which labels to attach to the container. 

After the container is created, it is started using: 

d.client.ContainerStart(...) 

If the container fails to start, the Agent removes it immediately: 

ContainerRemove(ctx, resp.ID, container.RemoveOptions{Force: true}) 

This prevents failed containers from being left behind. 

 

Host Networking 

The Worker container is started with host networking: 

NetworkMode: container.NetworkMode("host") 

This means the Worker container shares the host machine’s network stack. If the Worker listens on port 50051, it is reachable directly at: 

host-ip:50051 

This is important because the Master needs to connect to the Worker using gRPC. Host networking avoids extra Docker port-mapping complexity. 

For example, if the Agent advertises host: 

192.168.1.10 

and the Worker uses port: 

50051 

then the Worker address becomes: 

192.168.1.10:50051 

The Agent returns this address to the Master. 

 

Worker Address Returned to the Master 

After the container starts successfully, the Agent returns a CreateWorkerResponse: 

return CreateWorkerResponse{ 

    WorkerID:    workerID, 

    Address:     fmt.Sprintf("%s:%d", d.cfg.AdvertiseHost, hostPort), 

    HostPort:    hostPort, 

    ContainerID: resp.ID, 

} 

This response includes: 

Field 
	

Description 

WorkerID 
	

Unique Worker identifier 

Address 
	

Network address used by the Master to connect to the Worker 

HostPort 
	

The port assigned to the Worker 

ContainerID 
	

Docker container ID 

The Master uses the Worker address to create a gRPC client connection. 

 

 

Asynchronous Worker Readiness 

Starting a container does not guarantee that the Worker is immediately ready. The container may be running, but the Python gRPC server may still be loading. 

To handle this, the Agent waits until the Worker port is reachable before notifying the Master. 

This logic is in agent/handlers.go: 

go waitAndCallback(resp, callbackURL, agentID) 

The Agent checks the Worker port using: 

waitForPort(resp.Address, 100*time.Second) 

This repeatedly tries to open a TCP connection to the Worker address. 

If the port becomes reachable, the Agent sends a callback to the Master: 

POST /workers/ready 

The callback body contains: 

{ 

  "worker_id": "worker-id", 

  "address": "host:port", 

  "agent_id": "agent-id", 

  "container_id": "container-id" 

} 

Only after this callback does the Master register the Worker as available. 

This design avoids routing requests to Workers before they are ready. 

 

Container Labels and Recovery 

The Agent labels every Worker container with metadata: 

"llm.cluster.role": "worker" 

"llm.cluster.agent_id": d.cfg.AgentID 

"llm.cluster.worker_id": workerID 

"llm.cluster.host_port": strconv.Itoa(hostPort) 

These labels are very useful for recovery. 

When the Agent restarts, it can list all Docker containers and identify which ones belong to the cluster. This is implemented in: 

ListRunningWorkers() 

The Agent checks: 

    The container image matches the Worker image. 

    The label llm.cluster.role is worker. 

    The container state is running. 

    The Worker ID and host port labels exist. 

Then it rebuilds the Worker information and sends it to the Master during registration. 

This allows the Master to re-adopt running Workers instead of losing track of them after an Agent restart. 

Worker Cleanup and Removal 

The Agent also supports Worker removal. 

    Remove One Worker 

The /workers/destroy endpoint removes a specific Worker. It calls: 

DestroyWorker(workerID) 

The Agent finds the container by name: 

name := "llm-" + workerID 

Then it removes the container: 

ContainerRemove(ctx, name, container.RemoveOptions{Force: true}) 

This is used when the Master wants to remove a Worker, such as during scale-down. 

    Clean All Workers 

The /workers/clean endpoint removes all containers using the configured Worker image. This is implemented in: 

CleanWorkers() 

It lists all containers and removes the ones whose image matches the Worker image. 

This is useful for testing and resetting the environment. 

 

    Relationship With Autoscaling 

Docker-based deployment is directly connected to autoscaling. 

The Master’s autoscaler monitors cluster metrics such as: 

    Worker utilization 

    Queue depth 

    P95 latency 

    Requests per second 

    Number of healthy Workers 

When the Master decides that more capacity is needed, it selects an Agent and sends: 

POST /workers/create 

The Agent then creates a new Docker Worker container. 

When the system is under low load, the Master can mark Workers as draining and remove them after their active requests finish. 

So, Docker gives the system the ability to scale horizontally by adding and removing Worker containers at runtime. 

 

6.10 Monitoring and TUI Dashboard 

The project includes a real-time terminal-based monitoring dashboard for the Master node. This dashboard is used to observe the current state of the cluster while the system is running. The dashboard starts from the Master entry point in:

The Master launches the dashboard using:

This is important because the dashboard receives the same Router object used by the Master server. Therefore, the dashboard can read live cluster information directly from the Master’s in-memory state. 

The TUI implementation is in:

Purpose of the TUI Dashboard 

The monitoring dashboard is designed to make the distributed system easier to understand, debug, and operate. Since the system contains multiple components, such as the Master, Agents, Workers, Docker containers, ChromaDB, and Ollama, it is important to have one place where the system state can be observed. 

The dashboard helps the user monitor:

    Which Workers are healthy or unavailable.

    What errors, warnings, or important events occurred recently.

Dashboard Startup 

The dashboard starts when the Master process starts. In master/main.go, the main function creates a Router and starts the server:

    fmt.Printf("Error starting TUI: %v\n", err) 

}

    return err

7. Autoscaling and Resource Management 

Autoscaling is one of the critical capabilities of the LLM Cluster Orchestrator. The master component continuously evaluates cluster health and dynamically adjusts the number of active workers to balance cost efficiency with throughput and latency requirements. 

 

7.1 Autoscaling Problem 

LLM inference workloads are highly bursty. Traffic can increase by 10× within seconds—such as when many users submit prompts at once or a batch job starts—and then drop back to near zero. This makes static capacity planning inefficient. Provisioning for peak load wastes GPU and CPU resources during idle periods, while provisioning for average load leads to latency spikes, long queues, and dropped requests during traffic bursts. 

To address this, the system uses an autoscaler that continuously monitors key runtime signals (worker count, active requests, queue depth, latency, and error rate). Based on these signals, it adjusts the number of workers in real time. Cooldown windows are applied to prevent rapid oscillation, ensuring stable and efficient scaling behavior. 

7.2 Metrics Used 

The master polls all registered workers every heartbeat interval and aggregates the following metrics into a cluster-wide snapshot: 

 

Metric 
	

Source 
	

Why It Matters 

Worker count 
	

Worker registry in master 
	

Baseline — how many workers are currently HEALTHY 

Active requests 
	

In-flight tracker (inflight.go) 
	

Real-time load — requests currently being processed 

Queue depth 
	

Master request queue 
	

Backpressure indicator — requests waiting for a worker 

Latency (p50 / p95) 
	

Timing per request, aggregated 
	

User-experience signal — directly drives scale-up 

Error rate 
	

Failed vs total requests over sliding window 
	

Quality signal — high error rate may mean overloaded workers

showing live worker count, active requests, queue depth, and latency graphs 

7.3 Scale-Up Logic 

A scale-up event is triggered when any of the following thresholds are breached: 

    Queue depth exceeds the configured maximum queue depth threshold 

    p95 latency exceeds the latency SLO (e.g., 5 seconds) 

    Active requests per worker exceeds the concurrency ceiling 

    Error rate exceeds the error threshold over the sliding window 

 

When a scale-up is triggered, the master sends a spawn command to the agent. The agent starts a new worker process, which registers itself back with the master via gRPC. Until the worker completes registration and passes its first health check, it remains in the STARTING state and receives no traffic. 

 

Scale-Up Step 
	

Description 

1. Threshold breach detected 
	

Master evaluates metrics snapshot against thresholds 

2. Cooldown check 
	

Skipped if last scale-up was within the cooldown window 

3. Spawn command sent 
	

Master calls agent gRPC SpawnWorker RPC 

4. Agent starts worker 
	

Agent forks/starts the worker process on the local machine 

5. Worker registers 
	

Worker calls master RegisterWorker gRPC — state: STARTING 

6. Health check passes 
	

Master marks worker HEALTHY — now eligible for traffic 

 

7.4 Scale-Down Logic 

A scale-down event is triggered when ALL of the following conditions hold simultaneously — a more conservative check than scale-up to avoid premature removal of capacity: 

    Queue depth is zero (no backlog) 

    Active requests per worker is below the lower concurrency bound 

    p95 latency is comfortably below the SLO 

    Worker count is above the configured minimum 

 

The master selects the least-loaded worker and transitions it to DRAINING state. No new requests are routed to a draining worker. Once its in-flight requests complete (or the drain timeout elapses), the master sends a shutdown command and the worker exits cleanly. 

 

7.5 Cooldowns and Soft Thresholds 

Without cooldowns, the autoscaler can thrash — rapidly scaling up and down in response to transient spikes, which itself introduces latency and wastes resources. Two cooldown windows are enforced: 

 

Cooldown Type 
	

Purpose 

Scale-up cooldown 
	

Minimum time between consecutive scale-up events (e.g., 30 s). Prevents spawning many workers in response to a single burst before newly started workers have registered. 

Scale-down cooldown 
	

Minimum time between consecutive scale-down events (e.g., 60 s). Prevents removing workers too aggressively after a traffic drop. 

 

Soft thresholds add hysteresis: scale-up fires at queue depth > 10 but scale-down only fires when queue depth = 0 (not just < 10). This prevents oscillation around the boundary. 

 

7.6 Worker Lifecycle States 

Each worker in the master's registry passes through a defined state machine: 

 

State 
	

Meaning 
	

Receives Traffic? 

STARTING 
	

Worker process launched; awaiting registration + first health check 
	

No 

HEALTHY 
	

Registered and passing health checks; normal operation 
	

Yes 

DRAINING 
	

Marked for shutdown; completing in-flight requests only 
	

No (new requests only) 

DEAD 
	

Health checks failed or drain timeout elapsed; removed from registry 
	

No 

 

The state transitions are: STARTING → HEALTHY (on successful registration + health check), HEALTHY → DRAINING (on scale-down selection), DRAINING → DEAD (on drain completion or timeout), HEALTHY → DEAD (on missed health checks). 

 

7.7 Cost and Resource Efficiency 

The autoscaler is designed to be cost-aware. Running LLMs on GPU hardware is expensive — every idle worker represents wasted spend. The scale-down logic is intentionally aggressive about removing idle capacity while the cooldowns prevent the cost of thrashing. 

 

Efficiency Principle 

The system targets a steady-state worker utilization of 60–80%. Below 40% utilization triggers scale-down consideration. Above 85% active concurrency triggers scale-up. This band prevents both waste and overload. 

 

  

8. Fault Tolerance 

A production LLM cluster must handle failures gracefully — workers crash, networks partition, and hardware fails. The LLM Cluster Orchestrator implements multiple layers of fault tolerance to ensure requests are not silently dropped and the system recovers automatically. 

8.1 Worker Failure Detection 

The master detects worker failures through a combination of missed heartbeats and active health checks. Every worker sends a periodic heartbeat to the master. If a worker misses a configurable number of consecutive heartbeats, the master marks it as DEAD and removes it from the routing pool. 

Detection Mechanism 
	

Detail 

Heartbeat interval 
	

Workers ping master every N seconds (configurable) 

Miss threshold 
	

After K consecutive missed heartbeats, worker transitions HEALTHY → DEAD 

gRPC error detection 
	

Immediate failure if the gRPC stream to worker errors (connection refused, EOF) 

In-flight tracking 
	

inflight.go tracks all outstanding requests per worker; on worker death, in-flight requests are surfaced for retry or error response 

Screenshot of worker logs showing heartbeat messages being sent, or master logs showing health check results (PASS/FAIL) for each registered worker 

8.2 Health Checks 

Beyond heartbeats, the master performs active health checks by sending a lightweight ping RPC to each worker. This catches cases where the worker process is alive but the LLM inference engine has stalled 

8.3 Request Timeout Handling 

LLM inference can be slow. Without timeouts, a single slow model could hold a worker hostage indefinitely. The master enforces two timeout boundaries: 

    Request timeout: Maximum time allowed for a complete request/response cycle. Exceeded requests return a 504 Gateway Timeout to the client. 

    Drain timeout: Maximum time a DRAINING worker waits for in-flight requests to complete before being forcefully terminated. 

 

8.4 Slow Worker Handling 

A worker may be alive and healthy by heartbeat but processing requests abnormally slowly — for example, if the GPU is thermally throttling. The latency-based autoscaling trigger catches this: if p95 latency exceeds the SLO, the system scales out additional workers rather than waiting for the slow worker to finish. 

The routing strategy (round-robin or least-connections) naturally routes new requests away from the overloaded worker while it finishes its current load. 

 

8.5 What Happens if a Worker Dies Mid-Request 

If a worker dies while processing a request, the in flight tracker in the master detects the orphaned request when the worker's gRPC stream closes. The master returns an error response to the client. The current implementation does not automatically retry mid-flight requests, as LLM inference is not idempotent retrying could produce duplicate completions. 

Mid-request retries are not implemented. Clients are expected to implement retry logic with idempotency keys if needed.   

8.6 What Happens if the Master Dies 

The master is currently a single point of failure. If it crashes, workers continue running but cannot receive new routing assignments, and clients cannot submit new requests. The master is designed to restart quickly it persists state to SQLite so it can reconstruct the worker registry and in-flight state on restart. 

The README notes that restart can be handled by systemd or the kernel watchdog. A highly-available master (with leader election) is a future enhancement. 

Screenshot of master restart — show the server coming back up and re-registering workers from its persisted database state.  

8.7 Updating Workers Without Downtime 

Rolling updates are supported through the worker lifecycle. To deploy a new worker version: 

    Start new workers with the updated binary — they register as STARTING then HEALTHY 

    Drain old workers — transition them to DRAINING; they finish current requests 

    Old workers exit after drain completes — zero dropped requests 

 

This rolling approach means at least one old and one new worker overlap during the update window, ensuring continuous availability. 

8.8 Current Fault Tolerance Limitations 

 

Limitation 
	

Impact / Mitigation 

No mid-request retry 
	

Client sees error if worker dies mid-inference; client must retry 

Single master (no HA) 
	

Master crash = full outage until restart; mitigated by fast restart + system 

No cross-node agent coordination 
	

Each agent manages one host; no cross-host failover of workers 

No persistent request queue 
	

Queued-but-not-started requests are lost on master crash 

No circuit breaker 
	

A misbehaving worker receives traffic until heartbeat timeout elapses

11. Limitations 

While the system meets the project objectives, there are several limitations arising from environmental constraints and deliberate design scope decisions. 

 

11.1 Local Machine and Docker Limitations 

The project was developed and tested mainly in a local machine environment using Docker containers. This setup is useful for demonstrating the system architecture, worker deployment, request routing, RAG integration, and autoscaling behavior. However, it also introduces several limitations compared to running the system on a real distributed production cluster. 

Because the Master, Agent, Workers, ChromaDB, and Ollama may all run on the same physical machine during testing, the system does not fully represent the performance or reliability of a multi-machine deployment. 

 

Single-Machine Resource Constraints 

One major limitation is that all components may share the same machine resources. These resources include: 

CPU 

RAM 

Disk I/O 

Network stack 

Docker daemon 

Ollama runtime 

ChromaDB service 

In a real distributed system, Workers would usually run across multiple machines. Each machine would contribute additional CPU, memory, and possibly GPU resources. In the local setup, however, starting more Workers does not always mean adding real capacity. 

For example: 

1 Master + 1 Agent + 5 Workers 

may still be limited by the same local CPU, memory, and Ollama instance. 

This means local testing can show that the orchestration logic works, but it may not accurately represent the performance of a real cluster. 

 

Docker Overhead 

Docker makes Worker deployment easier and more consistent, but it also adds overhead. 

Each Worker container needs: 

    Container startup time. 

    Python runtime initialization. 

    gRPC server startup. 

    Environment variable setup. 

    Network binding. 

    Docker daemon management. 

Because of this, creating a new Worker is not instant. The Agent must create the container, start it, wait for the Worker gRPC port to become reachable, and then notify the Master through /workers/ready. 

This startup delay affects autoscaling. Even if the Master decides to scale up immediately, the new Worker may not be available until several seconds later. 

 

Host Networking Limitation 

The Worker containers use Docker host networking: 

NetworkMode: container.NetworkMode("host") 

This simplifies communication because the Worker can listen directly on the host’s port, and the Master can connect to it using: 

host:worker_port 

However, host networking has limitations: 

    It is less isolated than bridge networking. 

    Port conflicts are possible. 

    It may behave differently across operating systems. 

    It depends on the host network configuration. 

    It is less portable in some Docker environments. 

The Agent tries to reduce port conflicts by checking for available ports before creating Workers, but conflicts can still occur in edge cases if another process takes the port after the check. 

 

Port Range Limitation 

Workers are assigned ports from a configured range. By default, the Agent uses ports such as: 

50051 to 50150 

This means the number of Workers that can run on one Agent is limited by the available port range. 

If all ports in the range are used, the Agent cannot create more Workers and returns an error. 

This is handled in agent/docker.go by checking port availability, but it still creates a practical limit on local scaling. 

 

Shared Service Bottlenecks 

In the local setup, multiple Workers may share the same supporting services, especially: 

Ollama 

ChromaDB 

Docker daemon 

This creates possible bottlenecks. 

For example, even if the system creates several Worker containers, they may all call the same Ollama server through: 

OLLAMA_URL 

If Ollama is running only one model instance, then all Worker requests may still be bottlenecked by that shared inference backend. 

Similarly, ChromaDB may become a shared retrieval bottleneck if many Workers query it at the same time. 

Therefore, increasing Worker count does not always result in proportional performance improvement. 

 

Limited Fault Isolation 

Docker gives some isolation between Worker processes, but in a local setup the components still share the same physical machine. 

If the machine runs out of memory, experiences high CPU usage, or the Docker daemon fails, many components may be affected at the same time. 

For example: 

    The Master may become slow. 

    The Agent may fail to create Workers. 

    Existing Workers may become unresponsive. 

    ChromaDB may fail. 

    Ollama may time out. 

In a true distributed deployment, a failure on one machine would not necessarily affect the entire cluster. In the local setup, one machine failure can affect everything. 

 

Docker Daemon Dependency 

The Agent depends on the local Docker daemon to create and remove Worker containers. If Docker is not installed, not running, or unavailable, Worker deployment fails. 

This means the system requires: 

Docker installed 

Docker daemon running 

Permission to access Docker 

Worker image built locally 

If any of these requirements are missing, the Agent cannot create Workers. 

The Worker image must also exist locally under the configured name, such as: 

llm-worker:latest 

If the image is not built before running the Agent, Worker creation will fail. 

 

Startup Order Dependency 

The local system has several services that must be available in the correct order. 

For example: 

    ChromaDB should be running before Workers need RAG retrieval. 

    Ollama should be running before Workers perform inference. 

    The Agent must register with the Master before the Master can request Workers. 

    Workers must become ready before the Master can route traffic to them. 

The Agent tries to handle this by starting Ollama and ChromaDB during startup. However, startup delays can still happen, especially when Docker Compose or Ollama takes time to become ready. 

 

 

Difference From Real Cluster Deployment 

The local Docker-based setup is useful for development and demonstration, but it differs from a real cluster in several ways. 

Local Setup 
	

Real Cluster 

Components may run on one machine 
	

Components run across multiple machines 

Workers share local resources 
	

Workers may have dedicated resources 

Host networking is used 
	

Cluster networking or service discovery may be used 

Docker daemon is local 
	

Container orchestration may use Kubernetes or similar 

Manual image building may be required 
	

Images are usually stored in a registry 

Scaling is limited by local hardware 
	

Scaling depends on cluster capacity 

Failures affect many components 
	

Failures can be isolated per node 

Because of this, the local setup should be viewed as a functional prototype rather than a full production deployment. 

    All components run on a single local machine. 

    CPU, memory, and Docker overhead limit realistic scaling experiments. 

    Worker startup latency is higher than would be expected in a tuned production environment. 

 

11.2 GPU Simulation Limitations 

The project was designed as a distributed LLM cluster orchestrator, but the current implementation has limitations related to GPU simulation and GPU resource management. Since the system was tested mainly in a local or limited environment, the GPU behavior is not fully equivalent to a real production GPU cluster. 

In the current implementation, Workers are deployed as Docker containers and run local LLM inference through Ollama. However, the system does not directly manage or schedule real GPU resources. The Agent can create and remove Worker containers, but it does not yet check GPU availability, GPU memory usage, or GPU utilization before deciding where to place a Worker. 

 

No Real GPU Scheduling 

A major limitation is that the current Agent does not include GPU-aware scheduling. In agent/handlers.go, the /system/info endpoint reports basic machine information such as CPU usage, memory usage, and operating system information. However, it does not report GPU statistics. 

The current system reports values such as: 

    CPU usage 

    Memory usage 

    Operating system 

    But it does not report: 

    GPU utilization 

    GPU memory usage 

    Available VRAM 

    CUDA device count 

    GPU temperature 

    GPU process allocation 

Because of this, the Master cannot make placement decisions based on GPU availability. For example, if two Agents are registered, the Master can choose an Agent based on Worker count, but not based on which Agent has more free GPU memory. 

 

Simulation Does Not Fully Represent Real GPU Behavior 

If GPU behavior is simulated during testing, the simulation is limited because it cannot fully reproduce the behavior of real LLM inference on GPU hardware. 

Real GPU workloads are affected by many factors, such as: 

    Model size. 

    Available VRAM. 

    Number of concurrent requests. 

    Batch size. 

    Token generation speed. 

    GPU memory fragmentation. 

    Model loading time. 

    CUDA driver behavior. 

    Thermal throttling. 

    Other processes using the GPU. 

A simple simulation cannot accurately capture all of these factors. Therefore, performance results from the simulation should be treated as approximate and not as exact production measurements. 

 

No GPU Isolation Between Workers 

In the current Docker-based Worker deployment, Workers are created as containers, but the code does not explicitly assign specific GPUs to specific containers. 

For example, the Docker creation logic in agent/docker.go creates containers using: 

ContainerCreate(...) 

and configures host networking, restart policy, environment variables, and labels. However, it does not configure Docker GPU device requests such as: 

--gpus 

NVIDIA_VISIBLE_DEVICES 

CUDA_VISIBLE_DEVICES 

This means the current implementation does not isolate GPU access per Worker. If multiple Workers use the same Ollama instance or same GPU, they may compete for GPU resources. 

 

Shared Ollama Instance Limitation 

The Worker documentation states that the current approach uses a shared Ollama instance on the machine. The Agent starts Ollama and passes its URL to Workers using: 

OLLAMA_URL 

This means multiple Workers may send inference requests to the same Ollama server. 

While this is simpler and easier to run locally, it limits GPU simulation accuracy. In a real GPU cluster, each Worker may need a dedicated GPU, a dedicated model instance, or a carefully controlled sharing strategy. 

With a shared Ollama instance: 

    Workers do not truly own independent GPU resources. 

    Multiple requests may queue inside Ollama. 

    GPU bottlenecks may appear as Worker latency. 

    The Master may think multiple Workers exist, even though they share the same inference backend. 

    Scaling Workers may not improve performance if the shared GPU is already saturated. 

This is an important limitation because adding more Worker containers does not always mean adding more real compute capacity. 

 

Worker Count Does Not Equal GPU Capacity 

The autoscaler can create more Worker containers when load increases. However, if all Workers use the same machine and the same Ollama backend, increasing Worker count may not increase actual LLM throughput. 

For example: 

1 Worker + 1 Ollama instance + 1 GPU 

and: 

5 Workers + 1 Ollama instance + 1 GPU 

may still be limited by the same single GPU. 

This means Worker count is only an approximation of capacity. In a real GPU-aware system, capacity should be based on: 

    Number of GPUs. 

    Free GPU memory. 

    Model memory requirements. 

    Current GPU utilization. 

    Inference queue length. 

    Token generation throughput. 

 

Autoscaler Uses General Metrics, Not GPU Metrics 

The autoscaler currently uses cluster-level metrics such as: 

Worker utilization 

P95 latency 

Queue depth 

Error rate 

Requests per second 

RPS trend 

These are useful for general scaling, but they do not directly measure GPU load. 

For example, high latency may be caused by GPU saturation, but the system cannot confirm that because it does not read GPU utilization. Similarly, low Worker utilization may not mean the GPU is free if Ollama is busy with internal model execution. 

As a result, autoscaling decisions are based on indirect symptoms rather than direct GPU measurements. 

 

No Model Memory Awareness 

Different LLMs require different amounts of memory. A small model such as: 

smollm:135m 

requires far less memory than a larger model. 

The current implementation allows the model to be configured using: 

OLLAMA_MODEL 

However, the system does not estimate whether the selected model can fit into the available GPU memory. 

This means the system does not currently prevent cases where: 

    A model is too large for the GPU. 

    Too many Workers are created for the available VRAM. 

    Multiple model instances compete for memory. 

    Ollama fails because the model cannot be loaded efficiently. 

 

Limited Performance Accuracy 

Because GPU behavior is not directly measured or controlled, the system’s performance results are limited. Metrics such as response time and throughput are still useful, but they should be interpreted as system-level behavior, not exact GPU performance. 

For example, a stress test may show increasing latency under load. This indicates that the system is becoming overloaded, but it does not prove exactly whether the bottleneck is: 

    GPU inference time. 

    Ollama internal queueing. 

    Worker concurrency. 

    CPU usage. 

    Memory pressure. 

    Network overhead. 

    Docker overhead. 

Without GPU-specific metrics, the exact cause cannot be fully identified. 

    No physical GPUs were used. 

    Inference performance reflects CPU‑based execution rather than true GPU throughput. 

    Autoscaling behavior is correct, but absolute performance figures are not production‑representative. 

 

11.3 LLM Model Size Limitations 

The system uses a local LLM runtime through Ollama, and the default model configured in the Worker is: 

OLLAMA_MODEL = "smollm:135m" 

This is a very small model compared to modern large language models. It was selected because it is lightweight, easier to run locally, and suitable for demonstrating the project flow without requiring powerful hardware. 

However, using a small model introduces several limitations. 

 

Limited Answer Quality 

Smaller models usually have weaker reasoning and language generation capabilities compared to larger models. As a result, the generated answers may be: 

    Less detailed. 

    Less accurate. 

    More repetitive. 

    Less fluent. 

    More likely to misunderstand complex prompts. 

This means the project can demonstrate the orchestration and RAG pipeline, but the answer quality may not match larger LLMs. 

 

Reduced Knowledge Capacity 

Small models contain fewer parameters, so they store less general knowledge. The RAG pipeline helps by adding external context from ChromaDB, but the model still needs enough reasoning ability to use that context correctly. 

For example, even if the RAG system retrieves the correct document, a small model may still: 

    Ignore part of the context. 

    Summarize it poorly. 

    Mix retrieved context with incorrect assumptions. 

    Produce an incomplete answer. 

 

Context Length Limitation 

Local models often have limited context windows. This means they can only process a limited amount of text at one time. 

Because of this, the project limits the amount of retrieved RAG text using: 

RAG_MAX_CHARS 

and limits generated output using: 

num_predict: 200 

This keeps inference faster, but it also means the model receives only short snippets from the retrieved documents. If the answer requires multiple long documents, the model may not receive enough information. 

 

Performance vs. Quality Trade-Off 

The small model improves speed and makes the project easier to run on local hardware. However, this comes at the cost of response quality. 

A larger model could produce better answers, but it would require: 

    More RAM. 

    More GPU memory. 

    Longer startup time. 

    Higher inference latency. 

    More powerful hardware. 

Therefore, the selected model reflects a trade-off between practicality and output quality. 

 

Hardware Limitations 

The project is intended to run locally, so the model size must match the available hardware. If a larger model is used, the system may experience: 

    Slow response times. 

    Ollama timeouts. 

    High CPU or GPU usage. 

    Memory exhaustion. 

    Failed model loading. 

The Worker includes a timeout setting: 

OLLAMA_TIMEOUT_SECONDS = 120 

If the model takes too long to respond, the Worker returns a timeout message instead of crashing. 

 

Effect on Scalability Results 

Model size also affects scalability testing. With a small model, requests may complete faster, which makes the system appear more responsive. With a larger model, each request may take much longer, increasing queue depth and latency. 

Therefore, scaling results depend heavily on the selected model. The same cluster may behave very differently when using: 

smollm:135m 

compared to a larger model such as: 

llama3 

mistral 

gemma 

Because of this, performance results should always mention which model was used. 

    Relatively small local models were used for practicality. 

    Larger models would introduce:  

    higher memory usage, 

    longer startup times, 

    more pronounced scaling pressure. 

    These effects are not fully explored in this implementation. 

 

11.4 Single Master Limitation 

The current implementation uses a single Master node as the central controller of the system. The Master is responsible for receiving client requests, registering Agents and Workers, routing requests, tracking in-flight requests, collecting metrics, running the autoscaler, and displaying the monitoring dashboard. 

This design makes the system easier to implement and test, but it introduces an important limitation: the Master is a single point of failure. 

 

Master as the Central Coordinator 

In the current architecture, all major control decisions pass through the Master. 

The Master handles: 

    Client /chat requests. 

    Agent registration through /agents/register. 

    Worker readiness callbacks through /workers/ready. 

    Worker selection and routing. 

    In-flight request tracking. 

    Worker health checking. 

    Metrics collection. 

    Autoscaling decisions. 

    TUI dashboard state. 

Because all of these responsibilities are centralized, the system depends heavily on the Master remaining alive and reachable. 

 

Single Point of Failure 

If the Master process crashes or becomes unreachable, the cluster cannot operate normally. 

For example: 

    Clients cannot send new /chat requests. 

    Agents cannot register or report Workers. 

    New Workers cannot be added to the Master’s registry. 

    Worker health checks stop. 

    Autoscaling decisions stop. 

    The TUI dashboard stops updating. 

    In-flight request tracking is lost. 

Even if existing Worker containers are still running, the system cannot route new requests to them because routing is controlled by the Master. 

 

In-Memory State Limitation 

The Master stores important runtime information in memory. For example, the Router keeps maps of Workers and Agents: 

workers map[string]*Worker 

agents map[string]*AgentInfo 

It also keeps in-flight request tracking, metrics snapshots, autoscaler state, and startup statistics. 

Because this information is mostly in memory, if the Master restarts, it may lose: 

    Registered Workers. 

    Registered Agents. 

    Current in-flight requests. 

    Recent logs and events. 

    Recent metrics. 

    Autoscaling history. 

Some recovery exists through Agent registration. When an Agent restarts or registers again, it can report surviving Workers to the Master. However, this does not fully solve Master failure because the Master itself does not automatically restore all previous runtime state. 

 

No Leader Election or Replication 

The current implementation does not include multiple Master nodes. There is no leader election system, no replicated Master state, and no standby Master that can take over automatically. 

In a production distributed system, high availability is usually achieved using mechanisms such as: 

    Leader election. 

    State replication. 

    Distributed consensus. 

    Persistent shared storage. 

    Health-checked failover. 

    Load balancers in front of multiple controllers. 

This project does not currently include those mechanisms. Therefore, if the Master fails, another Master cannot automatically continue from the same state. 

 

Impact on Autoscaling 

Autoscaling is also centralized in the Master. The autoscaler runs inside the Master process and depends on metrics collected from the Router. 

If the Master goes down: 

    No new Workers can be created. 

    Low-load Workers cannot be removed. 

    Worker health changes are not detected. 

    Agent failures are not tracked. 

    Scaling cooldowns and history are lost. 

This means the cluster loses its self-management ability while the Master is unavailable. 

 

Impact on Monitoring 

The TUI dashboard is also part of the Master process. Since the dashboard is launched from master/main.go, if the Master process exits, monitoring stops as well. 

This means there is no independent monitoring service in the current implementation. Logs, events, and metrics are mainly stored in memory and shown through the Master TUI. 

 

Recovery Behavior 

The current design provides partial recovery through Agents and Workers. 

If Workers are still running after a Master restart, the Agent can report them during registration using the workers field. The Master can then reconnect to those Workers and re-adopt them. 

However, this recovery depends on: 

    The Agent being alive. 

    The Agent registering again. 

    The Worker containers still running. 

    The Master successfully reconnecting to each Worker. 

This is helpful, but it is not the same as full Master high availability. 

    The system relies on a single master node. 

    If the master fails:  

    worker state, 

    in‑flight request tracking, 

    autoscaler state are temporarily lost. 

    A production system would require master replication or leader election.

11.6 RAG Accuracy Limitations 

The project includes a RAG pipeline to improve LLM answers using external documents stored in ChromaDB. However, the current RAG implementation has several accuracy limitations. These limitations come from the size of the document set, the retrieval method, the amount of context passed to the model, and the ability of the LLM to correctly use the retrieved information. 

Although RAG improves the system compared to using the model alone, it does not guarantee that every answer will be correct. 

 

Limited Document Collection 

The system retrieves information from the documents stored in: 

vector-db/embedded-docs/ 

In the current implementation, these documents mainly contain sample World War II topics, such as D-Day, Pearl Harbor, Stalingrad, the Holocaust, and Allied/Axis powers. 

This means the RAG system can only provide useful context for topics that exist in the document collection. 

If the user asks about something outside the available documents, the RAG system may not retrieve helpful information. 

For example, if the user asks: 

Explain quantum computing. 

the current document set will not contain relevant information. In that case, the Worker may fall back to the model’s general knowledge, or it may retrieve unrelated documents. 

 

Simple Keyword-Based Retrieval 

Although the project uses ChromaDB as a vector database, the current Worker retrieval logic in worker/rag.py loads documents from ChromaDB and then ranks them using a simple keyword-count method. 

The scoring function counts how often prompt terms appear in each document: 

def _score(doc: str, terms: list[str]) -> int: 

    text = doc.lower() 

    return sum(text.count(t) for t in terms if t) 

This approach is simple and fast, but it has accuracy limitations. 

It may fail when: 

    The user uses different words than the document. 

    The question requires semantic understanding. 

    Relevant documents do not share exact keywords. 

    Common words accidentally increase the score of irrelevant documents. 

    The query is vague or short. 

For example, a query about “Normandy invasion” may be relevant to D-Day, but if the document uses different wording, keyword matching may not rank it correctly. 

A full semantic vector search would be more accurate because it would compare meaning, not only word overlap. 

 

Limited Context Size 

The retrieval system limits how much text is passed to the model. 

The configuration includes: 

TOP_K = int(os.getenv("RAG_TOP_K", "2")) 

MAX_CHUNK_CHARS = int(os.getenv("RAG_MAX_CHARS", "150")) 

By default, the Worker retrieves only: 

2 documents/snippets 

150 characters from each snippet 

This keeps the prompt short and inference faster, but it may remove important details. Some questions require more context than 150 characters per document. 

For example, if a document contains a long explanation, the retrieved snippet may include only the beginning and miss the specific detail needed to answer the question. 

 

No Advanced Chunking Strategy 

The initialization script stores whole documents in ChromaDB. The current implementation does not split documents into smaller semantic chunks before retrieval. 

This can reduce retrieval accuracy because: 

    Long documents may contain multiple topics. 

    Relevant information may appear in the middle or end of a document. 

    The retriever may truncate the beginning of the document and miss the useful part. 

    Smaller chunks usually provide more precise context. 

A more advanced RAG system would split documents into chunks, embed each chunk separately, and retrieve the most relevant chunks instead of whole documents. 

 

No Source Citation in Final Answer 

The Worker passes retrieved context to the model, but the final answer does not include source citations or document IDs. 

This means the user cannot easily verify which document was used to produce the answer. 

For academic or production use, it would be better to return: 

    The generated answer. 

    Retrieved document IDs. 

    Snippets used as evidence. 

    Similarity scores or ranks. 

This would make the system more transparent and easier to evaluate. 

 

Model May Ignore or Misuse Context 

Even when the correct context is retrieved, the LLM may not always use it correctly. 

The prompt tells the model: 

Use this context to answer the question. Give a direct answer only. 

However, small local models may still: 

    Ignore the context. 

    Misread the context. 

    Combine context with incorrect prior knowledge. 

    Generate unsupported details. 

    Produce incomplete answers. 

This is especially possible because the default model is small: 

smollm:135m 

Therefore, RAG accuracy depends not only on retrieval quality, but also on the model’s ability to understand and use the retrieved context. 

 

Fallback Can Return Generic Answers 

If RAG fails, the system returns an empty context: 

if not DOCUMENTS: 

    return "" 

Then the model is asked to answer directly. 

This keeps the system running, but it means the answer may no longer be grounded in the project’s documents. 

This can happen if: 

    ChromaDB is unavailable. 

    The collection is missing. 

    Documents fail to load. 

    No relevant documents are found. 

    The prompt does not match the document set. 

In those cases, the answer depends only on the LLM. 

 

No Retrieval Evaluation Metrics 

The current implementation does not measure retrieval accuracy. It does not calculate metrics such as: 

    Precision@K. 

    Recall@K. 

    Mean Reciprocal Rank. 

    Similarity score thresholds. 

    Answer faithfulness. 

    Context relevance. 

Because of this, it is difficult to quantitatively prove how accurate the RAG system is. Testing is mostly manual, using example queries and checking whether the retrieved context seems relevant. 

 

 

    RAG quality depends on the embedded document set. 

    The document corpus is limited and domain‑specific. 

    Retrieval errors or missing context can still lead to incomplete answers. 

    No re‑ranking or cross‑encoder validation is implemented. 

 

Reflection 

Despite these limitations, the system demonstrates the core principles of distributed LLM inference: request routing, autoscaling, failure handling, and separation of concerns. The limitations highlight realistic next steps rather than fundamental design flaws.

12. Conclusion 

This project demonstrates a practical distributed computing solution for high-concurrency LLM inference. It separates external client access from internal worker execution, uses a master/load balancer to coordinate routing, uses agents to manage worker processes, and uses Python workers to combine RAG with local model inference. The architecture is appropriate for the coursework problem because it applies core distributed-system ideas: component separation, concurrency, scheduling, queueing, monitoring, fault tolerance, and elastic scaling. 

 The strongest part of the design is that it treats LLM inference as a cluster workload rather than a single process function call. Requests are tracked, workers have lifecycle states, metrics are smoothed, and autoscaling decisions are based on several signals rather than one raw value. The remaining limitations are realistic for a prototype and provide clear paths for future work, especially master high availability, GPU-aware scheduling, stronger security, and controlled benchmark reporting.

C.3 Worker Selection Strategy 

File: 

master/lib/strategy.go 

func (r *Router) pickLeastConnections() (*Worker, error) { 

    var best *Worker 

    var minActive int64 = -1 

 

    for _, worker := range r.workers { 

        if !worker.isRoutable() { 

            continue 

        } 

 

        active := worker.ActiveRequests() 

        if best == nil || active < minActive { 

            best = worker 

            minActive = active 

        } 

    } 

 

    if best == nil { 

        return nil, ErrNoWorkersAvailable 

    } 

 

    return best, nil 

} 

Explanation: 
This code selects the healthy Worker with the fewest active requests. It helps distribute requests more efficiently across Workers.

C.5 Worker gRPC Handler 

File: 

worker/main.py 

def Handle(self, request, context): 

    priority = request.priority if request.priority in (0, 1) else 1 

    tier = "pro" if priority == 0 else "free" 

 

    self.queue.push(priority, request.request_id, request.message) 

 

    item = self.queue.pop() 

    if item is None: 

        context.abort(grpc.StatusCode.INTERNAL, "queue unexpectedly empty") 

        return 

 

    _, req_id, message = item 

 

    rag_context = retrieve(message) 

 

    reply = run_model( 

        prompt=message, 

        context=rag_context, 

        worker_port=WORKER_PORT 

    ) 

 

    depth = self.queue.size() 

 

    return worker_pb2.Response( 

        request_id=req_id, 

        reply=reply, 

        queue_length=depth, 

    ) 

Explanation: 
This code receives a request from the Master, adds it to the priority queue, retrieves RAG context, calls the LLM, and returns the generated answer. 

 

C.6 RAG Retrieval 

File: 

worker/rag.py 

def retrieve(prompt: str, top_k: int = TOP_K) -> str: 

    if not DOCUMENTS: 

        return "" 

 

    terms = [t for t in prompt.lower().split() if len(t) > 2] 

 

    scored = sorted( 

        ((doc, _score(doc, terms)) for doc in DOCUMENTS), 

        key=lambda x: x[1], 

        reverse=True, 

    ) 

 

    top_docs = [doc[:MAX_CHUNK_CHARS] for doc, s in scored[:top_k] if s > 0] 

 

    if not top_docs: 

        top_docs = [doc[:MAX_CHUNK_CHARS] for doc in DOCUMENTS[:top_k]] 

 

    context = "\n\n".join( 

        f"[{i+1}] {doc}" for i, doc in enumerate(top_docs) 

    ) 

 

    return context 

Explanation: 
This function retrieves relevant document snippets based on the user prompt and returns them as context for the LLM.
